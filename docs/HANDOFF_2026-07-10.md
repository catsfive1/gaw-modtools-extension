## What was fixed this session

### 1. Profile "eater" (`/u/catsfive` / `/u/me` posts) — the RIVER was scrambling post order
**Symptom (reported across 12+ sessions):** the profile *posts* tab showed only the newest post, then "jumped to 3–4 days old" — recent posts looked eaten.

**Root cause (found by inspecting the LIVE DOM via Claude-in-Chrome, NOT by reading code — that's what every prior session got wrong):** *nothing hides or removes the posts.* The profile **river** (`enhanceUserProfilePage` / `loadNextProfilePage`, modtools.js) appends fetched pages **out of chronological order** into fragmented `.post-list` containers, so the operator's recent posts (#2 .. ~3 days) get stranded **below** older 4–6-day posts. Proof: a raw `fetch('/u/catsfive/?type=submission')` returns correct newest-first order (`19h,19h,20h,21h,1d…`) while the live DOM showed `19h,4d,4d,4d…`. Only the client DOM is scrambled.

**Fix:** `_reorderProfilePostsChronological()` + a debounced MutationObserver wired into `enhanceUserProfilePage` — re-sorts every `.post[data-id]` by its `<time>` descending into the first `.post-list` (catches both the river's appends and GAW's native scroll). Idempotent, self-limiting. Test: `scripts/_profile_river_reorder_smoke_test.mjs` (5/5). **LIVE-VERIFIED:** posts now list `13h · 15h · 18h · 1d · 1d · 1d · 1d · 3d · 3d…` in correct order.

> **Note — the v10.36.10 "aux detector parity" fix was a RED HERRING.** A 12-agent adversarial trace proved every in-file hide/remove path is already gated off on profile tabs (and the upvote-age filter's polarity is *inverted* vs the symptom). That fix is a real latent bug (the CSS-veto safety net was disarmed on owner tabs) so it was left in (harmless, shipped) — but it is **not** the eater. The **river ordering** is.

### 2. "/users not working + other things broken" — autoRefresh reload-storm (v10.40.2)
**Root cause (found via a stubbed boot harness + reading the tick):** `autoRefreshTick` (`setInterval(…, 60000)`) reloaded **any unfocused/background GAW tab on EVERY 60-second tick** — the gate `const hidden = document.hidden || !document.hasFocus(); const idle = (now-lastActivity) >= intervalMs; if (!hidden && !idle) return;` had **no time gate on the `hidden` branch** (only `idle` did). So the instant the operator alt-tabbed to the chat, every GAW tab — including `/users` — reloaded once a minute: Triage Console never finished mounting ("/users not working"), page state was lost ("other things broken"), and even automated inspection tabs died on load. v10.40.0's settings-integrity hydrate activated `autoRefreshEnabled` (default true), lighting up the latent bug.

**Fix:** `if (!idle) return;` — reload only after a full N-minute idle interval; an unfocused tab still reaches idle (no activity events fire), so intended behavior holds + self-limits. Test: `scripts/_p19_autorefresh_idle_gate_smoke_test.mjs` (7/7). **LIVE-VERIFIED:** `/users` Triage Console mounts — `#gam-triage` present, 50 rows, no mount-fail banner, no console errors.
