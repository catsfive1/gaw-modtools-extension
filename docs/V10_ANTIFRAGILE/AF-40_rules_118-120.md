## Rule 118 -- Self-Diagnostics on Every Update

### Current state

`chrome.runtime.onInstalled` fires on install and update. The existing handler
(background.js L216) does three things: logs the reason, calls `_recordSwBoot`,
and recreates all 7 alarms. It runs no diagnostic checks. On `reason='update'`
the extension is silently live with no verification that the new version's
storage schema, token state, or RPC indexes are intact.

There is a Maintenance panel (popup.js) with 12 manual routines covering quota,
token age, diag log rotation, and cookie clearing. These are click-only. No
subset runs automatically on update.

### Gap

A version bump can introduce a new storage key shape, rename an existing key,
or change the format of `gam_diag_log`. If the update lands while the user is
modding, they will see silent failures -- wrong data in the panel, stale cached
intel, an RPC that returns 404 because the endpoint was renamed -- with no
actionable surface until a human opens the popup and manually runs a routine.

### Proposed diag suite (design only, not implemented here)

Add a `_runUpdateDiagnostics()` function called from `onInstalled` when
`details.reason === 'update'`. Four probes, each independently try-caught so
one failure does not abort the others. Results accumulate into a
`gam_diag_update_<version>` key in `chrome.storage.local`. On any failure,
write a `gam_maint_warning` entry (the same channel the existing maintenance
alarms use) so the existing popup Maintenance panel surfaces it automatically.

**Probe 1 -- Schema integrity.** Read `gam_settings` and verify the top-level
shape contains the keys the current version expects. Any missing required key
triggers a `setSetting` backfill with the default value and logs a WARNING (not
an error -- missing keys are recoverable). A key whose stored value is the
wrong type (e.g., a string where an object is expected) is logged as ERROR and
surfaced in the Maintenance banner.

**Probe 2 -- Token validity.** Call the existing `tokensStatus` path (already
used by the popup on open). If both tokens are absent, log INFO -- this is
normal for a fresh install. If at least one token is present but fails the
`__isValidTokenOrEmpty` regex, log ERROR and add a Maintenance warning:
"Update detected stale token shape -- re-enter your token in the popup."

**Probe 3 -- Indexes still work.** Attempt a `chrome.storage.local.get` for
the three most-read compound keys: `gam_profile_intel`, `gam_parked_items`,
`gam_diag_log`. Verify each is either absent or matches its expected shape
(object with known top-level fields). A parse failure means the stored value
was written by an older version with an incompatible schema. Log ERROR and
write a Maintenance warning with a specific repair action ("Open Maintenance
panel -> Routine 6: Purge diag log" or "Routine 11: Schema migration").

**Probe 4 -- Critical RPCs respond.** After a 2-second delay (to let the SW
fully settle post-install), fire `ping` to itself via `chrome.runtime.sendMessage`.
This is already used by the popup. If `ping` fails or times out after 5 s,
log ERROR: "SW self-ping failed post-update." This catches the rare case where
the update left the background script in a broken state.

**Maintenance banner integration.** The `gam_maint_warning` key is already
read by popup.js and rendered as a red banner at the top of the Maintenance
tab. The update diag probes plug into the same channel, so no new UI code is
needed. The banner message should include the specific repair action, not just
"something failed."

**What this does not do:** it does not block the extension from loading, does
not prompt the user automatically, and does not auto-repair. All repair actions
are click-only in the existing Maintenance panel. The diag runs silently and
surfaces findings only when the user next opens the popup.

## Rule 119 -- Feature Removability

### Method

For each named feature, I traced all symbols that (a) the feature registers,
renders, or writes, and (b) any other feature that reads or depends on those
symbols. The dependency edge is the removability cost.

### Feature inventory and dependency map

**Hot Now Panel**

Entry point: `_showHotNowPanel()` / `_closeHotNowPanel()` in modtools.js.
Triggered by the SIREN button (`gam-siren-count`) in the mod toolbar.

Dependencies in: reads `gam_profile_intel` (shared with Intel panel), reads
`gam_parked_items` to show parked status per-user, calls `rpcCall('modSearch')`
and `rpcCall('bugReportList')` (shared RPCs).

Dependencies out: nothing else reads Hot Now state. The SIREN button exists
independently; removing the panel would leave the button inert unless removed
with it.

Removability verdict: **CLEAN.** Deleting `_showHotNowPanel`, `_closeHotNowPanel`,
and the `sirenBtn.addEventListener('click', _showHotNowPanel)` wire-up removes
the feature without breaking anything. The SIREN button count logic (which reads
`susMark` state for the badge number) is separate and can be retained or removed
independently. No other feature calls into Hot Now.

**Modmail 3-column layout**

Entry point: `mmIs3Col` flag computed from `window.innerWidth >= 1280` at
panel build time (modtools.js L15416). Controls CSS class on the panel root.

Dependencies in: none -- it reads only viewport width.
Dependencies out: the 3-col CSS rules are scoped to `.gam-modmail-3col` selector
class. Nothing outside the modmail panel reads or sets that class.

Removability verdict: **CLEAN.** Remove the `mmIs3Col` branch and force the
single-column layout path. Zero cross-feature impact.

**AI Hold Queue (Parked Items)**

Entry point: `gam_parked_items` CS store (`_ST.parked`), `modParkedCreate` RPC,
`gam-park-btn` buttons injected per-post.

Dependencies in: `gam-park-btn` is injected by the post/comment action bar
builder. `_ST.parked` is read by Hot Now Panel to show parked state per-user.

Dependencies out: the `gam-parked` CSS class is applied to the host DOM
element for visual dimming; no other feature reads this class.

Removability verdict: **CONDITIONAL.** Removing AI Hold Queue also breaks the
parked-user indicator in Hot Now Panel (which reads `_ST.parked.entries`).
If both are removed together, it is clean. If only AI Hold Queue is removed
while Hot Now Panel is retained, Hot Now must be updated to remove the parked
state display. This is a two-line change, not a systemic dependency. Flag it
in any removal PR.

**Universal Undo**

Entry point: `_undoSlot` variable and associated middleware (modtools.js L6451).
Wraps the ban RPC path -- on a successful ban, stores the ban metadata in
`_undoSlot` for a configurable TTL window. An "Undo" affordance appears in the
post-ban confirmation snack.

Dependencies in: reads nothing external. The slot is in-memory only; it does
not write to `chrome.storage`.

Dependencies out: nothing reads `_undoSlot` except the undo affordance render
path and the ban confirmation handler. The ban RPC itself (`modPost('ban')`)
is not gated on `_undoSlot` -- removing undo does not affect ban execution.

Removability verdict: **CLEAN.** The slot and the undo affordance can be
deleted without any cross-feature impact. The ban path continues to function.
One caveat: the undo affordance is rendered inside the ban confirmation snack;
removing it requires touching the snack render function, but that is a small
contained change.

**Tier System**

No explicit tier-gating symbols found in modtools.js. The `adminModPromote`
RPC handler in background.js validates `tier` against allowlist
`['mod','senior_lead','lead']` (confirmed in AF-12 audit). The tier data comes
from the worker's modprofile response and is consumed by the toolbar badge
renderer and the modmail UI for display purposes only.

Dependencies in: reads the `tier` field from modprofile RPC response. If the
field is absent, the UI falls back to blank gracefully (no defensive code
required -- the badge simply does not render).

Dependencies out: no other feature branches on tier in client code. Tier is
display-only on the client; all tier-gated server behaviour is enforced
worker-side.

Removability verdict: **CLEAN for client display.** Removing the tier badge
from the toolbar is a CSS/DOM-only change with no cross-feature breakage.
Removing tier validation from `adminModPromote` is a worker-side concern
outside this codebase.

### Feature flag proposal

Current state: no feature flag system exists. Features are compiled in.

Proposed: add a `FEATURE_FLAGS` constant object at the top of modtools.js,
defaulting all features enabled. Each flag guards its feature's entry point
only -- not every internal call site.

```js
// Proposed addition -- modtools.js, near top of IIFE
const FEATURE_FLAGS = Object.freeze({
  HOT_NOW_PANEL:      true,
  MODMAIL_3COL:       true,
  AI_HOLD_QUEUE:      true,
  UNIVERSAL_UNDO:     true,
});
```

Guarding pattern: one conditional at the wire-up point, not scattered
throughout the implementation.

```js
// Hot Now -- existing line:
sirenBtn.addEventListener('click', _showHotNowPanel);
// Guarded:
if (FEATURE_FLAGS.HOT_NOW_PANEL) sirenBtn.addEventListener('click', _showHotNowPanel);

// Universal Undo -- existing entry:
// wrap ban RPC call
// Guarded: the _undoSlot middleware block is wrapped in:
if (FEATURE_FLAGS.UNIVERSAL_UNDO) { /* slot capture + affordance */ }

// AI Hold Queue -- park button injection:
if (FEATURE_FLAGS.AI_HOLD_QUEUE) { /* inject gam-park-btn */ }
```

Disabling `HOT_NOW_PANEL` via the flag with `AI_HOLD_QUEUE` enabled is a
supported state because Hot Now's parked-indicator dependency is in the panel
code, not in the queue itself. The queue continues to accumulate; Hot Now just
does not show it.

Tier system has no flag because there is no client-side code to gate: the
badge render is already conditional on the field being present in the API
response.

## Rule 120 -- Bug Regression Test Infrastructure

### Current state

No `tests/` directory exists in the repository. Bug reports are tracked in the
worker's `/bug-reports` table (confirmed by `bugReportList` and `bugReportUpdate`
RPCs). There is no convention for associating a closed bug report with a
code-level regression test.

### What was built

Created `tests/regressions/` directory and `tests/regressions/README.md`
(at `D:\AI\_PROJECTS\modtools-ext\tests\regressions\README.md`).

The README establishes:

- **Naming convention:** `<bug-id>.test.js` where bug-id is the numeric `id`
  from the worker's bug-reports table.
- **Minimum file structure:** describe block names the bug ID, one `it` block
  reproduces the failing condition.
- **Four enforcement rules:** one file per bug, bug ID in the describe string,
  runnable in isolation, tests are never deleted (only annotated as superseded).
- **Severity comment mandatory:** LOW / MEDIUM / HIGH, defined explicitly.
- **Workflow gate:** no bug report row should be marked `closed` without a
  corresponding file committed in the same PR.

### What remains to build before tests can run

The regression tests reference `chrome.*` APIs. A chrome-mock stub does not yet
exist. Before any test file can be executed:

1. **Choose a test runner.** Jest with jsdom is the natural fit for a
   browser-extension codebase without a bundler. `jest.config.js` would set
   `testEnvironment: 'jsdom'` and add a global setup file that stubs
   `chrome.storage`, `chrome.runtime`, `chrome.alarms`.

2. **Minimal chrome-mock stub.** Located at `tests/support/chrome-mock.js`.
   Must cover: `chrome.storage.local.get/set/remove`, `chrome.runtime.sendMessage`,
   `chrome.runtime.onInstalled` (event emitter pattern), `chrome.alarms.create/get`.
   The stub should be synchronous-by-default (returns resolved promises) to keep
   test code readable.

3. **First regression test to write.** The first real regression test should
   target an already-closed, well-understood bug. A good candidate is the
   `lsSet` clobber (gam_settings overwritten by a stale-blob write) which was
   fixed and is concrete enough to write a deterministic test: mock `storage.local`,
   call `setSetting` twice with different keys, assert neither write clobbers the
   other. This test would have caught the original regression.

### Connection to AF-36 testing roadmap

AF-36 (see `AF-12_rules_34-36.md`) proposed a circuit breaker for
`chrome.runtime.sendMessage`. The regression test for any future circuit-breaker
bug (OPEN state not resetting, probe logic error) belongs in this directory.
The convention established here applies to all three files -- background.js,
modtools.js, popup.js -- not just content-script logic.
