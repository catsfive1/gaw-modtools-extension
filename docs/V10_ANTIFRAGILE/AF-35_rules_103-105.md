## Rule 103 — Test Every Flow: Network Offline, Permissions Denied, SW Killed

### Current State: Zero Formal Coverage

There are no unit tests and no E2E tests. Resilience exists in the code (AbortController 15s timeouts, chrome.runtime.lastError guards, safeGet fallbacks, multi-source CSRF recovery) but it is entirely unverified by automated harness.

---

### Proposed Minimum Viable Chaos Checklist

A chaos runbook that can be executed manually per release and eventually automated.

**Network Offline**
- [ ] Open /modqueue. Go offline (DevTools > Network > Offline). Click Ban on any item. Expected: snack "request timed out" within 15s; no spinner stuck; panel dismisses cleanly.
- [ ] Open Mod Console mid-workflow. Kill network. Submit. Expected: error message surfaced; no unhandled rejection in console; panel remains open so mod can retry.
- [ ] Trigger Deep Analysis scan with network offline. Expected: progress bar terminates with error state; no partial badge corruption.
- [ ] Trigger auth rehydrate (popup > Rehydrate button) while offline. Expected: failure snack; auth state not corrupted (mod token still readable after reconnect).
- [ ] Open modmail panel. Kill network mid-thread-load. Expected: "loading..." does not freeze; timeout fires; list shows error row.

**Permissions Denied (403/401 from worker)**
- [ ] Worker returns 401 for workerFetch RPC. Expected: rpcCall surfaces "permission denied" snack; does not retry infinitely; does not clobber secretCache.
- [ ] /admin/import-tokens-from-kv called by non-lead mod. Expected: popup shows readable error, does not hang.
- [ ] CSRF token absent (no cookie, no meta, no hidden input). Expected: apiCore falls through all four sources, surfaces "CSRF unavailable" rather than sending a request with an empty token.
- [ ] chrome.storage.local.set() denied mid-write (quota exceeded). Expected: setSetting catches, logs to _diagBuffer, does not throw uncaught exception.

**Service Worker Killed**
- [ ] SW evicted between popup open and Rehydrate click. Expected: secretCache falls back to chrome.storage.local (confirmed in loadSecrets fallback path); popup shows stale-token warning, not a crash.
- [ ] SW killed between setTokens (content script) and next workerFetch. Expected: background message handler reloads secrets from session/local storage on the next call rather than sending empty tokens.
- [ ] Alarm fires (gam_maint_quota_check) while SW is dead. Expected: alarm re-wakes SW; handler completes; no duplicate alarm creation on re-wake.
- [ ] Extension updated mid-session (new SW activates). Expected: gam_update_available flag written to storage; update banner appears on next page load; no auto-reload, no RCE surface.

### Proposed Playwright Script Scope (no code — scope only)

**File:** `tests/e2e/chaos-auth-wizard.spec.ts`

**Prerequisites:** Chromium with extension loaded via `--load-extension`; a test account with a known mod token seeded into storage via a beforeAll fixture.

**Test 1 — Happy path baseline**
Load `/modqueue`. Confirm the extension boots (status bar present, `window.__GAM_MT_LOADED === true`). Confirm clicking a ban button opens the Mod Console panel. Confirm ESC closes it cleanly.

**Test 2 — Network offline during ban submit**
Intercept all fetch calls via `page.route('**', route => route.abort())` before clicking Submit. Assert: snack with text matching "timeout" appears within 20s. Assert: no uncaught console errors. Assert: `page.evaluate(() => document.querySelector('#gam-mc-panel'))` returns null (panel closed or visible error state, not frozen).

**Test 3 — 401 from worker during RPC**
Mock worker endpoints to return 401. Trigger rehydrate. Assert: popup status element contains error text. Assert: no infinite retry loop (no more than 2 network calls per user action, verified via `page.route` call count).

**Test 4 — Storage permission denied**
Before extension init, call `chrome.storage.local.clear()` via extension messaging. Then navigate to `/modqueue`. Assert: extension does not throw; fallback memory-get path activates (verify via `_diagBuffer` containing `safeGet` entries).

**Test 5 — SW killed between actions**
Use `chrome.runtime.reload()` (test-only) to evict SW between two popup interactions. Assert: second popup open recovers tokens from `chrome.storage.local`; no "no runtime" error surfaces to UI.

**Coverage not in scope for MVP:** Deep Analysis batch flows, modmail AI reply, auto-DR sweep. Those require seeded D1 data and a live worker mock — v.next.

## Rule 104 — "Fix Everything" Button

### Current State

The popup has 12+ individual maintenance buttons (Trim now, Clear intel cache, Rotate diag log, Run maintenance, etc.) spread across the Maintenance section. A mod encountering a broken state must know which button to push. There is no single-shot recovery path.

The `rehydrateBtn` (auth re-hydrate) and the maintenance routines are completely separate UI flows with no orchestration layer between them.

---

### Proposed: Single "Fix Everything" Button

**Location:** Top of the Maintenance section in `popup.html`, above the individual routine buttons.

**Label:** `Fix Everything`

**Sequence (runs in order, stops on unrecoverable failure):**

1. **Storage repair** — enumerate `chrome.storage.local`, remove any key with a corrupt (non-parseable) value, trim `gam_diag_log` to last 200 entries, trim `gam_profile_intel` to entries younger than 48h. Mirrors what `gam_maint_diag_rotate` and `gam_maint_intel_evict` alarms do autonomously, but synchronous and user-triggered.

2. **Auth re-hydrate** — call the same RPC that `rehydrateBtn` calls: verify mod token against worker, refresh `gam_settings` in both `chrome.storage.session` and `chrome.storage.local`, update `secretCache` in the background SW. Surface result (token age, team membership) inline.

3. **Cache invalidation** — clear `gam_profile_intel` (hover-card cache), clear `gam_learned_selectors` (force fresh selector discovery on next page load), clear any stale `gam_update_available` flag if the currently installed version matches worker `/version`.

4. **Diag log purge** — write an empty array to `gam_diag_log` in both `chrome.storage.local` and the in-memory `_diagBuffer`. Log a single synthetic entry marking the purge timestamp and the triggering button.

**Progress UI:**

- Replace button label with a progress indicator: `[1/4] Storage repair...` -> `[2/4] Auth re-hydrate...` etc.
- Each step gets a checkmark or red X inline.
- Final state: `All 4 repairs complete` (green) or `Completed with 1 error — see step 2` (amber).
- No spinner-only UI; each step result must be readable without waiting for the whole sequence.

**Implementation note (not code):** The four sub-routines already exist as discrete functions triggered by individual buttons. The "Fix Everything" handler is a thin orchestrator that calls them in sequence, collects per-step results, and renders progress. No new repair logic is needed — only the orchestration wrapper and the progress UI strip.

## Rule 105 — Native Browser Behavior Violations

### Finding 1: Modmail Slide-In Panel Does Not Lock Scroll — Correct, But Incomplete

The modmail panel (`_showModmailPanel`) sets `position:fixed; top:0; right:0; bottom:0` and slides in via `transform:translateX`. It does NOT set `document.body.style.overflow = 'hidden'`. This means the page behind the panel remains scrollable via keyboard (arrow keys, Page Down, Space) while the panel is open.

**Verdict: partial violation.** Mouse wheel over the main content area is blocked by the fixed overlay, but keyboard-initiated scroll (Space, Page Down, arrow keys) still scrolls the underlying page. On close, scroll position is not explicitly restored. Since no `scrollY` is saved before open and no `window.scrollTo` is called on close, the browser's native scroll position is preserved by default — but only if the user did not keyboard-scroll while the panel was open.

**Recommendation:** On panel open, save `window.scrollY`. On panel remove (both ESC handler at line 15358 and click-handler at line 15354), call `window.scrollTo(0, savedY)` to guarantee position restoration regardless of what happened during panel lifetime.

---

### Finding 2: history.pushState / popstate Interception — Back Button Risk

Lines 377-381 monkey-patch `history.pushState` and `history.replaceState`, and listen to `popstate`. The wrappers call `_handleNav` via `setTimeout(..., 150)`. This interception is required for SPA nav detection — but introduces a correctness risk:

If `_handleNav` throws (or calls code that throws) after the browser has already committed the popstate navigation, the extension's internal nav-state diverges from the browser's actual URL. The `panelOpen` variable and mounted DOM elements (triage console, status bar, gam-update-banner) may reflect the wrong page.

**Current mitigation:** `_handleNav` has no try/catch of its own. If any sub-call throws, the nav state is silently corrupted.

**Verdict: latent violation.** Not currently causing reported user-facing breaks, but the pattern will cause back-button confusion in any scenario where `_handleNav` throws mid-SPA-nav.

**Recommendation:** Wrap `_handleNav` body in try/catch. On catch: log to `_diagBuffer`, call `closeAllPanels()` as a safe reset, and set `panelOpen = null`. Do not re-throw.

---

### Finding 3: Escape Key Capture — Interferes With Browser-Native ESC Behaviors

`closeAllPanels` is wired to `keydown Escape` at line 10595 via a document-level listener with `e.preventDefault()`. This is correct for modal dialogs, but the modmail panel installs its OWN escape handler (line 15358) that calls `panel.remove()` without `e.preventDefault()` and does not check whether another panel is already open.

Two listeners both fire on ESC when both the modmail panel and a `.gam-modal` are open simultaneously (edge case: modmail panel open + bug-report modal triggered). The modmail handler fires first (added later in the event queue), removes the panel, then `closeAllPanels` fires and attempts to sweep `.gam-modal` nodes — safe but redundant.

The `e.preventDefault()` in the global handler suppresses any browser-native ESC behavior (e.g., stopping media playback, closing browser-native dialogs). On a page where ESC has native meaning (a browser `<dialog>` or a `<video>` in fullscreen), the extension consumes the event first.

**Verdict: violation when browser-native ESC targets exist on the page.** The wins.win platform currently has no `<dialog>` or fullscreen video that we observe, so this is dormant.

---

### Finding 4: Page-Down Behavior on Queue Pages

The status bar (`#gam-status-bar`) is `position:fixed; bottom:14px`. The modmail hints panel (`#gam-mm-hints`) is `position:fixed; right:12px; top:30%`. Neither element interferes with scroll mechanics, but the status bar occupies ~42px of fixed real estate at the bottom. On short viewport heights, the last queue item's action strip can sit behind the status bar and require extra scrolling to reach.

**Verdict: UX degradation, not a native-behavior break.** Page-Down still moves the document scroll the full viewport height; no `preventDefault` is called on scroll events. No violation, but worth noting.

---

### Finding 5: closeAllPanels Does Not Restore Scroll Position

Every panel that calls `closeAllPanels` (Mod Console, Intel Drawer, Hot Now, Help, Settings, Log) exits without restoring `window.scrollY`. Individual panels (modmail, intel drawer) also do direct `.remove()` without scroll restoration.

For panels opened by clicking a post-level action button (ban button, note icon), the page may have been at a specific scroll position. After the panel closes, if the user keyboard-scrolled while the modal was open (which the extension does not block), they land at a different position with no way back except manual scrolling.

**Verdict: minor native-behavior deviation.** Not a back-button break, but conflicts with the browser convention that an overlay closing returns the user to where they were. Fix: save/restore `scrollY` on all panel open/close pairs.
