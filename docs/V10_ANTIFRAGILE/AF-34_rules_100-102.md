# AF-34 Audit: Rules 100-102 (Anti-Fragile Suite)
**GAW ModTools v10.5.1 -- AUDIT-ONLY**
**Date:** 2026-05-09

## Rule 100 -- Export / Import (Data Portability)

### Export: Status PASS

Export is fully implemented. `exportBtn` lives in the popup footer (`popup.html:616`).
The handler (`popup.js:433-452`) scopes the export to `OWNED_KEYS` only, calls
`scrubExport()` to strip `workerModToken`, `leadModToken`, and `gam_sniff_log` before
writing, and downloads a dated JSON file. The payload includes `exportedAt`, `version`,
`scope: 'owned-keys'`, and the key list. Clean.

### Import: Status MISSING

No `importBtn` exists anywhere in `popup.html` or `popup.js`. There is no handler,
no file-picker, no re-apply logic. A mod who wipes their data (or migrates a browser)
has no way to restore from their export without manually injecting JSON into DevTools.
This contradicts Rule 100's intent -- export without import is read-only backup, not
portability.

### Proposal: `importBtn`

Place `importBtn` adjacent to `exportBtn` in the footer, separated by the existing
`pop-sep` dot. Handler flow:

1. Open a hidden `<input type="file" accept=".json">` and prompt the user to select
   a previously-exported ModTools JSON.
2. Parse the file. Reject immediately if `payload.scope !== 'owned-keys'`.
3. **Schema-version compatibility check:** compare `payload.version` against the current
   manifest version. If the exporting version's `MAINT_SCHEMA_KEY` value (readable from
   the payload's `gam_settings[MAINT_SCHEMA_KEY]`) is older than `MAINT_SCHEMA_CURRENT`
   (currently `3` -- `popup.js:3657`), run the existing additive migration logic from
   `maintRunSchemaCheck` on the imported settings object before applying. If the stored
   schema version is *newer* than current code, warn the mod and abort (downgrade risk).
4. Never overwrite `workerModToken` or `leadModToken` from an import -- those keys are
   device-bound and must be re-entered manually. Show a notice.
5. Write the sanitized payload via `chrome.storage.local.set()`. Show a snack
   confirming which keys were restored, which were skipped.
6. Reload the popup.

**What NOT to do:** do not import `gam_sniff_log` (may contain auth payloads). Do not
import `gam_fallback_mode` (device state). Both should be on a skip-list matching the
existing `scrubExport()` exclusions.

## Rule 101 -- Universal Undo Coverage

### What exists

Two distinct undo systems ship in v10.5.1:

**System A (persistent stack, `_UNDO_KEY`, `modtools.js:4559`):** 10-entry ring buffer
in `chrome.storage.local`. Handles `dr-add` only (death row queue entry via user
click). Ctrl+Z hotkey. Status-bar button. The `undoLastModAction()` dispatcher has an
explicit `else` branch that rejects unknown types with a snack -- so any future action
type wired here is additive and safe.

**System B (`withUndo` middleware, `modtools.js:6554`):** in-memory single-slot
(replaced per action), 20s Tier A / 5s Tier B window, toast-with-progress-bar UX,
`U`-key hotkey. Currently wraps `apiBan` only (`modtools.js:8557-8577`).

### Actions audited for undo coverage

| Action | Destructive? | Covered? | System | Gap |
|---|---|---|---|---|
| Ban | Yes | YES | System B (`withUndo`) | None |
| Unban | No (restorative) | n/a | -- | Not needed |
| Post/comment remove (`apiRemove`) | Yes | NO | -- | UNCOVERED |
| Watchlist add (`toggleWatch`) | Soft | NO | -- | UNCOVERED |
| Watchlist remove (`toggleWatch`) | Yes | NO | -- | UNCOVERED |
| DR add (user-action) | Yes | YES | System A | None |
| DR remove (`removeFromDeathRow`) | Yes | NO | -- | UNCOVERED -- see below |
| Macro delete (`__macroDelete`) | Yes | NO | -- | UNCOVERED |
| Sticky toggle (`apiSticky`) | Soft | NO | -- | UNCOVERED |
| Full clear (`clearAll`) | Critical | NO | -- | Triple-confirm only |

### Detail on each gap

**`apiRemove` (post/comment remove):** Called from the NBA panel (`modtools.js:5598-5599`)
with no wrapping. Remove is fired and logged but the inverse (`/approve` via
`apiApprove`) is available -- this is a natural `withUndo` candidate (Tier B, 5s).
The action is irreversible to the end-user if they misclick REMOVE vs APPROVE.

**Watchlist add/remove (`toggleWatch`):** `toggleWatch` is a pure toggle over
`lsGet/lsSet` (no network call), so the inverse is just calling `toggleWatch` again.
Three call sites: the Quick panel button, the mod-log watchlist section X button, and
the `W`-key hover shortcut. None use `withUndo`. Given it's local-only storage, even
a 5s System B toast would suffice. The snack already fires; it just needs an Undo
button.

**DR remove (`removeFromDeathRow`):** Two call sites -- the mod log DR section cancel
button (`modtools.js:9646`) and the Hot Now panel cancel (`modtools.js:9798`). Neither
uses `withUndo` or `_recordUndoAction`. The inverse is straightforward: re-push the
death row entry (username + reason + executeAt) back via the existing `addToDeathRow`
path. This should be System A (persistent, since DR data is important) wired as a new
type `'dr-remove'` alongside the existing `'dr-add'`.

**Macro delete (`__macroDelete`, `popup.js:3191`):** Currently behind a native
`window.confirm()`. The operation is synced to the team worker, so the inverse requires
a re-create call to `macroCreate`. Given the team-wide impact, this warrants at minimum
a System B toast (5s), ideally System A with the macro payload stored in the undo
entry. The existing `window.confirm()` guard is not sufficient -- it's a synchronous
modal that blocks the UI and has no undo path after confirmation.

**Sticky toggle (`apiSticky`):** Called in the NBA panel (`modtools.js:5601`). The
`/sticky` endpoint is documented as a toggle -- calling it again reverses the action.
This is the lowest-risk candidate for a System B toast (Tier B, 5s) since the inverse
is identical to the action. Note: `autoUnstickyEnabled` is disabled by default
(`modtools.js:1293`) due to the toggle-endpoint fragility bug, so undo here would be
purely manual.

### Recommendation: extend `withUndo`

All gaps except DR remove fit cleanly into System B (`withUndo`). Proposed call pattern
for each:

```
// apiRemove (post/comment) -- NBA panel
await withUndo(
  () => apiRemove(thingId, thingType),
  { tier: 'B', label: 'Removed ' + thingType,
    inverse: () => apiApprove(thingId, thingType) }
);

// toggleWatch -- all 3 call sites
withUndo(
  () => toggleWatch(username),
  { tier: 'B', label: wasWatched ? username + ' unwatched' : username + ' watched',
    inverse: () => toggleWatch(username) }
);

// apiSticky -- NBA panel
await withUndo(
  () => apiSticky(thingId),
  { tier: 'B', label: 'Sticky toggled',
    inverse: () => apiSticky(thingId) }
);
```

DR remove should use System A (`_recordUndoAction` + new dispatcher branch in
`undoLastModAction`) since it is persistent and session-crossing. Macro delete
needs the macro payload captured before deletion and re-posted on undo -- that
requires a `macroCopy` snapshot before the RPC call.

## Rule 102 -- Progressive Disclosure

### Current state: Maintenance card PASS

The Maintenance card (`popup.html:155-272`) correctly applies progressive disclosure:
four mod-friendly self-heal routines visible by default; six advanced/diagnostic
routines folded into a `<details class="pop-maint-advanced">` accordion labelled
"System diagnostics (advanced)". This is exactly the pattern Rule 102 calls for.

### Lead card: PARTIAL

The Lead card (`card-lead`) wraps its advanced settings behind
`<details id="leadSettingsAccordion">` (popup.html:418), labelled "Settings &
Maintenance (advanced)". Collapsed by default. That accordion contains: Flag expiry
input, Bug reports panel, bug visibility config, and four lead-tier maintenance
buttons. This is correct progressive disclosure.

However, the Lead card body above the accordion (always visible) already shows:
- Lead token input + Save
- Invite link button
- Mod rotation roster button

That is a reasonable visible set (3 items). No bloat issue here.

### Mod Console tabs: PASS (no stuffing)

The Mod Console (`openModConsole`) renders five tabs: Intel, Ban, Note, Message, Quick.
Each tab renders only its own panel -- no tab surfaces more than its scoped controls.
Intel tab is the richest (account summary, mod history, AI sidebar check, modnote) but
these are all in-scope for an intel view. No progressive disclosure issue.

### Tools card: LOW-PRIORITY CONCERN

The Tools card (`card-tools`, `popup.html:131-153`) shows 5 buttons always-on:
Debug snapshot, Dashboard, and 3 crawl buttons (10-page, 30-page, queue). The crawl
buttons in particular are power-user operations -- a new mod clicking "Crawl /users
(30)" by accident triggers a 30-page network sweep. Recommendation: consider collapsing
the Data harvest row into a secondary accordion within the Tools card, leaving only
"Debug snapshot" and "Dashboard" visible. Not a critical gap, but the Maintenance card
set the right precedent.

### Popup footer: CONCERN (will worsen with import)

The popup footer currently holds two items: `exportBtn` ("Export log") and `clearBtn`
("Clear all"). `clearBtn` is triple-confirmed before executing -- adequate. When
`importBtn` is added (Rule 100), the footer will hold three items. At that density,
"Clear all" (destructive, red) next to "Import" (restorative, neutral) next to "Export"
(neutral) creates a dangerous spatial proximity. Recommendation: move `clearBtn` into
the Maintenance card advanced accordion (where it conceptually belongs -- it's a reset
operation). The footer becomes export/import only, both non-destructive.

### Lead area density: WATCH

The Lead card currently hides `#leadOnlyTools` until `__applyLeadGate` confirms
`is_lead`. Non-leads see only the token input. Leads see the full card including the
accordion. The lead accordion contains 4 subsections (Team settings, Bug reports,
Bug visibility, Lead maintenance). That is starting to feel dense -- if a fifth
subsection is added, a nested accordion or tab split should be considered.
