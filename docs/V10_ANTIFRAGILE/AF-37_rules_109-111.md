## Rule 109 -- Automatically Bump Storage Version on Every Breaking Change

### Current Version Landscape

The codebase has two parallel schema tracking mechanisms, which is itself the first finding:

**Mechanism A -- Page localStorage schema** (`modtools.js`)
- Constant: `SCHEMA_VERSION = 2` (line 4309)
- Key: `gam_schema_version` (localStorage, page domain)
- Runner: `runMigrations()` (line 4352), called once at content-script init
- Migrations coded: v1 (roster `pending` -> `new`), v2 (`hideSidebar` forced ON)

**Mechanism B -- Extension settings schema** (`popup.js` + `background.js`)
- Constant: `MAINT_SCHEMA_CURRENT = 3` (popup.js:3657, background.js:660)
- Key: `gam_settings_schema_version` stored **inside** the `gam_settings` object in `chrome.storage.local`
- Runner: `maintSchemaCheck()` (popup.js:4012), triggered from the Maintenance panel's "Run all" routine
- Migrations coded: additive only -- installs missing keys from `MAINT_DEFAULT_SETTINGS`

The two mechanisms track different things (page-domain localStorage shape vs. extension settings object shape), which is architecturally correct. The naming divergence (`gam_schema_version` vs. `gam_settings_schema_version`) is a maintenance hazard -- future devs will not immediately understand the split.

### Audit: Recent Breaking Changes vs. Schema Bumps

| Change | Version Introduced | Breaking? | Schema Bumped? | Verdict |
|---|---|---|---|---|
| Roster status `pending` -> `new` | Pre-v8 (migration 1) | YES -- old clients reading `status === 'pending'` miss entries | YES -- `SCHEMA_VERSION` 0 -> 1 | PASS |
| `hideSidebar` default flip ON | v8.2.2 (migration 2) | Soft -- existing `false` values honoured; only missing-key case affected | YES -- `SCHEMA_VERSION` 1 -> 2 | PASS |
| `features.platformHardening` added as required key | Post-v8 | YES -- `validateSettingsShape` fails if absent; AF-07 repair patch covers it | YES -- `MAINT_SCHEMA_CURRENT` bumped to 3, key added to `MAINT_DEFAULT_SETTINGS` | PASS |
| `gam_ai_suspect_queue` introduced (CS wrapper, `aiSuspect` cloud store) | v10.x | Additive -- new key, no existing key renamed or removed | NO -- but additive changes don't require bumps | PASS (additive) |
| Multi-lead `isLeadMod` flag added to `gam_settings` | v5.x | YES -- code gates UI on `isLeadMod()`; older settings objects missing key silently default to `false` | MAINT_SCHEMA_CURRENT already covers this via `REPAIR_REQUIRED_SHAPE` + AF-07 repair | CONDITIONAL PASS -- repair path covers it but no explicit migration block documents the intent |
| `leadModToken` string field added | v5.x | YES -- empty string default is safe but the field must exist for token validation | Same as above -- covered by shape repair | CONDITIONAL PASS |
| `gam_shadow_decisions` + `gam_parked_items` cloud stores added | v8+ | Additive -- new keys, not renames | NO -- additive | PASS |
| `ai_hold_queue` D1 table (migration 032) | v11 spec | NOT a storage-layer breaking change (server-side schema, not extension localStorage/storage.local) | Worker migration sequence (032) handles this independently | N/A |

**Finding: no unambiguous missed bump.** The two breaking changes that got conditional passes (`isLeadMod`, `leadModToken`) are covered by the AF-07 shape-repair path introduced in v10.5.1, but they were never formalized as numbered migration blocks in `runMigrations()`. This creates a documentation gap: someone reading `runMigrations()` sees only two explicit migrations and cannot tell whether the settings shape changes were ever intentionally versioned.

### Critical Gap: Schema Version Split Is Undocumented

`SCHEMA_VERSION = 2` in modtools.js and `MAINT_SCHEMA_CURRENT = 3` in popup.js are **not the same thing and do not track the same artifact**. There is no comment, no doc, no `MIGRATION_LOG` explaining why they differ. A maintainer will eventually assume they should match and either panic or introduce a wrong-direction fix.

### Proposed: `schemaVersionGuard` Helper

This is the enforcement layer that makes Rule 109 automatic rather than relying on discipline.

```js
// ── schemaVersionGuard (add to modtools.js, near runMigrations) ──────────
//
// Call at the TOP of any PR that touches gam_settings shape, K.ROSTER shape,
// or any other localStorage/storage.local key the extension reads at boot.
//
// Usage (in a migration block or unit test):
//   schemaVersionGuard({ expected: 3, actual: SCHEMA_VERSION, context: 'migration 3' });
//
// Throws loudly in dev (non-prod) when the caller's expected version does not
// match the code constant. Silent no-op in prod so a stale guard never crashes
// a mod's browser.
//
function schemaVersionGuard({ expected, actual, context }) {
  if (actual === expected) return;
  const msg = '[ModTools schemaVersionGuard] VERSION MISMATCH in ' + (context || '?') +
    ': expected SCHEMA_VERSION=' + expected + ', found ' + actual +
    '. Did you increment SCHEMA_VERSION and add a migration block?';
  // Loud in dev builds; warn-only in prod to avoid crashing mods.
  if (typeof __GAM_DEV_BUILD !== 'undefined' && __GAM_DEV_BUILD) {
    throw new Error(msg);
  } else {
    console.error(msg);
    try { _diagLog('schemaVersionGuard', 'mismatch', { expected, actual, context }); } catch(_){}
  }
}

// Expose for test harness.
try { window.__GAM_schemaVersionGuard = schemaVersionGuard; } catch(_){}
```

**How the guard enforces Rule 109:**

When a developer adds a new key to `gam_settings` (or renames one), they add a migration block inside `runMigrations()`. That block calls `schemaVersionGuard({ expected: N+1, actual: SCHEMA_VERSION, context: 'migration N+1' })` before executing. If they forgot to increment `SCHEMA_VERSION`, the guard throws in dev and emits a diag error in prod. The bump is not optional; the code enforces it.

**Dual-mechanism alignment rule** (add as a code comment, not an enforcement block since the two constants track different artifacts):

```
// MIGRATION DISCIPLINE NOTE (v10.5.1 AF-37):
// SCHEMA_VERSION (this file, page localStorage) and MAINT_SCHEMA_CURRENT
// (popup.js / background.js, chrome.storage.local gam_settings) are SEPARATE
// version counters tracking separate storage artifacts. They INTENTIONALLY
// differ. Do NOT merge them. Do NOT assume they should be equal.
// -- Page schema governs: K.ROSTER shape, K_SCHEMA key, localStorage keys
// -- Settings schema governs: gam_settings object key presence and types
// Bump the relevant counter when adding/renaming/removing keys in its domain.
```

## Rule 110 -- Migration Tests in the Test Suite

### Current State

There is no automated test suite for the extension. All verification is manual (maintenance panel routines, diag log inspection, smoke-test scripts). This is the baseline AF-36 already identified.

### Required Tests: Schema Migration Property Tests

For every transition `version_N -> version_N+1`, the test must verify:

1. A storage object at version N, after `runMigrations()` or `maintSchemaCheck()`, equals the expected version-N+1 object.
2. The migration is **idempotent** -- running it twice produces the same result as running it once.
3. **No data loss** -- keys present in the N-state that are not part of the migration are still present in the N+1 state.
4. **Corrupt input tolerance** -- a storage object with a null/missing version field runs migrations from scratch without throwing.

### Proposed Test Layout

The test runner should be a standalone Node.js file (no browser required) that mocks the localStorage and chrome.storage.local APIs. Vitest or plain Node assert both work; pick based on whatever build step is eventually adopted.

```
tests/
  migrations/
    helpers.js                    # mock lsGet/lsSet, chrome.storage.local stub
    page-schema.test.js           # tests for runMigrations() (SCHEMA_VERSION track)
    settings-schema.test.js       # tests for maintSchemaCheck() (MAINT_SCHEMA_CURRENT track)
    schema-version-guard.test.js  # tests for schemaVersionGuard() helper
```

**`page-schema.test.js` -- required cases:**

```js
// T1: v0 -> v2 (full chain from clean install)
// Setup: localStorage empty (simulates brand new extension).
// Run: runMigrations()
// Assert: gam_schema_version === '2'
// Assert: no roster entries mutated (nothing to migrate)

// T2: v1 -> v2 (v1 client gets the v8.2.2 hideSidebar flip)
// Setup: gam_schema_version = '1'; gam_settings.hideSidebar = false
// Run: runMigrations()
// Assert: gam_schema_version === '2'
// Assert: getSetting('hideSidebar') === true  [migration 2 fired]

// T3: Idempotency -- v2 re-run
// Setup: gam_schema_version = '2'; gam_settings.hideSidebar = true
// Run: runMigrations() twice
// Assert: gam_schema_version === '2' (not '4')
// Assert: hideSidebar still true

// T4: Roster migration v0 -> v1 (pending -> new)
// Setup: K.ROSTER = { user1: {status:'pending'}, user2: {status:'new'} }
// Run: runMigrations()
// Assert: user1.status === 'new'; user2.status === 'new'

// T5: Corrupt version field
// Setup: gam_schema_version = 'NaN'
// Run: runMigrations()
// Assert: does not throw; gam_schema_version === '2' at end
```

**`settings-schema.test.js` -- required cases:**

```js
// T6: v1 -> v3 (missing keys installed)
// Setup: gam_settings = { workerModToken:'tok', isLeadMod:false }
// gam_settings.gam_settings_schema_version = 1
// Run: maintSchemaCheck()
// Assert: schema version in gam_settings === 3
// Assert: features.platformHardening === true  [MAINT_DEFAULT_SETTINGS]
// Assert: workerModToken === 'tok'  [existing keys preserved -- no data loss]

// T7: v3 -> v3 (already current -- no-op)
// Setup: gam_settings with version=3 + all required keys present
// Run: maintSchemaCheck()
// Assert: chrome.storage.local.set NOT called  [no unnecessary write]

// T8: Downgrade guard (v5 stored, v3 code)
// Setup: gam_settings.gam_settings_schema_version = 5
// Run: maintSchemaCheck()
// Assert: returns warn status; does NOT overwrite to v3

// T9: validateSettingsShape -- missing key detection
// Run: validateSettingsShape({ autoRefreshEnabled: true })  [9 keys missing]
// Assert: result.ok === false
// Assert: result.missing.length === 9

// T10: validateSettingsShape -- mistyped key
// Run: validateSettingsShape({ ...fullValidObject, tardsThreshold: 'two' })
// Assert: result.ok === false; result.mistyped === ['tardsThreshold']
```

**`schema-version-guard.test.js` -- required cases:**

```js
// T11: matching versions -- no-op
// T12: mismatch in dev build -- throws Error with descriptive message
// T13: mismatch in prod build -- console.error + diagLog, no throw
```

### Linking to AF-36

AF-36 established that the general test harness does not yet exist. These migration tests are the first module to build once a runner is wired. They have zero browser dependencies (all storage calls are mockable with a Map) and should be the zero-cost baseline that proves the test infrastructure works before adding UI or network tests.

## Rule 111 -- Canary Releases via Update URL

### Distribution Reality

ModTools is **not on the Chrome Web Store.** Distribution is via Drive Desktop sync: Commander pushes a new `.zip` into the shared Google Drive folder; Drive syncs it to each mod's machine; the mod manually reloads the unpacked extension via `chrome://extensions`. The v10.5.0 Discord DM flow (BUILDER-DISCORD, migration 035, `discord_dm_log`) added `POST /admin/dist/push-zip` to R2 and `POST /admin/rotation/dm-all-unrotated` to send signed download URLs per mod. The `discord_dm_log` table records delivery per mod (columns: `recipient_mod`, `zip_version`, `zip_url`, `http_status`).

There is no CWS `update_url` and no auto-install mechanism. Every update is manual-reload after manual-download. The canary mechanism must work within these constraints.

### Proposed Canary Mechanism

**Phase 1 -- Designate canary mods.**

Add a boolean flag `is_canary` to the `bot_mods` table (worker-side). Two mods volunteer as canaries -- ideally the two most technically comfortable with the extension. Flag set via a new lead-only endpoint or directly via D1 console.

```sql
-- Migration 036: canary flag
ALTER TABLE bot_mods ADD COLUMN is_canary INTEGER NOT NULL DEFAULT 0;
-- Designate initial canaries:
UPDATE bot_mods SET is_canary = 1 WHERE mod_username IN ('canary_mod_1', 'canary_mod_2');
```

**Phase 2 -- Staged R2 ZIP upload.**

The existing `POST /admin/dist/push-zip` endpoint writes to `dist/extension/v<version>.zip`. Add a parallel R2 key path for canary builds: `dist/extension/canary/v<version>.zip`. The lead uploads the canary ZIP to the canary path first, full-release path later.

**Phase 3 -- Canary-aware DM endpoint.**

Extend `handleAdminDmAllUnrotated` with a `canary_only: true` body flag. When set, the endpoint:

1. Queries `bot_mods WHERE is_canary = 1`.
2. Generates a signed R2 URL from the canary key path (7-day TTL, same as existing flow).
3. DMs only those mods via Discord with the canary ZIP link and a clear `[CANARY BUILD]` label in the message body.
4. Writes `discord_dm_log` rows per canary mod with `zip_version = 'v<X+1>-canary'`.

**Phase 4 -- Crash/error rate monitoring.**

The existing `gam_diag_log` (stored in `chrome.storage.local`, rotated by the `gam_maint_diag_rotate` alarm) already captures errors from the content script. The `maintDiagStatus` popup routine (Maintenance panel, Routine #6) reads this log and surfaces error counts. After canary distribution:

1. The two canary mods run the extension for 24-48 hours.
2. Lead runs "Maintenance > Full run" from the popup, copies the diag JSON to clipboard.
3. Lead pastes the JSON into a Discord thread (or back to Commander). The `discord_dm_log` table provides proof of delivery timestamps for the canary mods, so the observation window is unambiguous.

**Decision gate:** if the diag log from both canary mods shows zero `[ERROR]` or `[WARN]` entries attributable to the new version, the lead runs `POST /admin/rotation/dm-all-unrotated` with the full-release ZIP and `canary_only: false`. If errors appear, the canary ZIP is held and a patch is cut.

**Phase 5 -- Logging the rollout in `discord_dm_log`.**

The `discord_dm_log` table already has `zip_version` and `recipient_mod` columns. After full rollout, the lead can query:

```sql
SELECT recipient_mod, zip_version, http_status, ts
  FROM discord_dm_log
 WHERE kind = 'rotation_invite'
 ORDER BY ts DESC;
```

This gives a per-mod delivery audit trail showing which mods received canary vs. full-release and when. No new table or schema change required for Phase 5 -- `discord_dm_log` is sufficient.

**What "clean" means for the gate:**

- Zero `[ERROR]` diag entries with a v<X+1> version tag in the `v` field.
- Zero auth failures (no `auth-wizard-reopen` events in the log).
- Maintenance panel "Run all" exits with all green for both canary mods.

**Why this is the right mechanism for Drive Desktop distribution:**

A CWS `update_url` would be cleaner but requires CWS submission (pending). Until CWS is live, Drive Desktop + Discord DM is the actual delivery path. The canary mechanism proposed above uses only infrastructure that already exists: `discord_dm_log`, `bot_mods`, R2, and the existing DM endpoint. The only new artifact is the `is_canary` column and the `canary_only` flag on the DM endpoint -- both are low-risk additive changes.
