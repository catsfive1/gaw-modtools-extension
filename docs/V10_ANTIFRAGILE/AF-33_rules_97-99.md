# AF-33: Anti-Fragile Audit -- Rules 97-99
**GAW ModTools v10.5.1 | AUDIT-ONLY | 2026-05-09**

---

## Rule 97 -- Never require a full page reload to fix extension issues

### Finding: auth-fail banner calls `location.reload()` on success (VIOLATION)

**Location:** `modtools.js:21431-21432`

```js
btnRetry.textContent = 'Success -- reloading';
setTimeout(() => location.reload(), 400);
```

The "Force re-hydrate" button in `__showAuthFailBanner` follows this path on success:

1. `preloadSecrets()` -- pulls fresh token from `chrome.storage.local` into `_secretsCache`
2. `syncSecretsToBackgroundVault()` -- pushes to SW vault
3. `__validateModAuth()` -- verifies token against `/mod/whoami`
4. If `re.ok`: sets button text to "Success -- reloading", then fires `location.reload()` after 400ms

The reload is unnecessary. By the time step 3 succeeds, `_secretsCache` is fully hydrated and the SW vault is synced -- the state that `init()` needs is already in place. A bare `init()` call at this point is functionally equivalent to a reload, without the cost.

**Why it's a violation:** The banner exists precisely because the user is mid-session and hit an auth interruption. Forcing a full reload destroys any unsaved state on the page (open mod consoles, partially filled ban forms, modmail thread position) and adds ~2-4s of full page repaint. Rule 97 says never.

**Proposed fix -- in-place re-init:**

```js
if (re && re.ok) {
  btnRetry.textContent = 'Auth restored -- reloading UI...';
  try { b.remove(); } catch(_){}
  try { await init(); } catch(e){
    // init() is idempotent for most subsystems. If it partially fails,
    // the mod is at least unblocked -- surface the error as a snack.
    try { snack('Partial re-init: ' + (e.message || e), 'warn'); } catch(_){}
  }
}
```

**Feasibility assessment -- listener leak risk:**

`init()` is not currently idempotent by design, but the actual listener-leak surface is narrow:

- `installSpaWatcher()` -- already guards with `if (window.__GAM_SPA) return;` -- safe to call twice.
- `ModChat.init()` -- unknown guard; would need a `window.__GAM_MODCHAT_INIT` sentinel or a `try/catch` wrapper.
- `buildTriageConsole()`, `enhanceBanPage()`, `enhanceQueuePage()` -- all check for existing DOM nodes before inserting. Safe.
- `installSniffer()` -- would need a sentinel, or the existing try/catch absorbs a benign duplicate.

Net verdict: in-place re-init is feasible. The two items without sentinels (`ModChat.init`, `installSniffer`) need a one-line guard each. That is a v10.6 patch, not a blocker.

**Second violation: onboarding modal (line 19734)**

```js
try { setTimeout(() => { try { init(); } catch(e){} }, 400); } catch(e){}
```

This path -- token save in the onboarding modal -- already calls `init()` directly. So the precedent exists and works. The auth-fail banner is the only remaining call site using `location.reload()` on the success path.

**Third violation: line 21165 (SPA watcher?)**

Line 21165 is `location.reload()` -- context needed:

```
21165:    location.reload();
```

This is inside the `storage.onChanged` listener that fires when `gam_settings` changes. If the SW restarts and triggers a storage change, this reloads the page unconditionally. This is a secondary Rule 97 violation; not in scope of this audit item but flagged for v10.6.

## Rule 98 -- Persist user progress across extension restarts

Three surfaces audited: macro drafts, modmail AI drafts, wizard step.

### Surface 1: Macro draft (canned response memory, v10.5)

**Store used:** `chrome.storage.session` (key: `gam_macro_drafts`)

`chrome.storage.session` is cleared on browser quit and on browser tab close. It survives:
- SW restart (service worker restart does not touch session storage)
- Extension disable/enable cycle: **NO** -- disabling the extension clears `chrome.storage.session`

Survival matrix:

| Scenario | Survives? |
|---|---|
| SW restart | YES |
| Browser tab close + reopen (same session) | YES (session storage is per-profile, not per-tab) |
| Browser quit (Chrome fully closed) | NO -- session storage cleared |
| Extension disable/enable | NO -- session storage cleared |
| Extension update reload | NO -- session storage cleared |

**Verdict:** Partial compliance. SW restarts are the most common failure mode and are handled. Browser quit loses drafts -- this is a Rule 98 violation for any macro draft longer than a few words. The fix is to mirror `gam_macro_drafts` to `chrome.storage.local` with a TTL key, falling back to session on read.

### Surface 2: Modmail AI drafts (v9.x, pre-drafted replies)

**Store used:** `chrome.storage.session` (key: `gam_modmail_drafts`)

Same survival matrix as macro drafts. Same verdict: SW restart safe, browser quit not.

Additional issue: these drafts are generated by a background crawl of `/modmail/recent`. They are expensive (4 parallel AI calls, ~3-5s per thread). Losing them on browser quit means the crawl re-runs from scratch on next session. For a mod checking modmail after closing and reopening Chrome, all pre-drafted replies are gone.

**Verdict:** Rule 98 violation on browser quit. Lower severity than macro drafts because the content can be regenerated, but the 3-5s regeneration latency is user-visible friction. Fix: promote to `chrome.storage.local` with a 4-hour TTL. Stale drafts beyond TTL are silently dropped and regenerated.

### Surface 3: Wizard step (ban page enhancer)

The ban page enhancement (`enhanceBanPage`, line 13837) is a DOM decorator -- search filter + unban buttons injected into the `/ban` page. It has no multi-step wizard. There is no in-progress state to persist; the function is stateless UI scaffolding that runs fresh on each page load/SPA nav.

**Verdict:** No Rule 98 violation here. No wizard step exists. The audit brief assumed a wizard -- it does not exist in v10.5.1. Closest candidate is the onboarding modal (token claim flow), which is single-shot and correctly clears itself on completion.

## Rule 99 -- Show helpful recovery steps for common failure modes

### Current banner copy audit

`__showAuthFailBanner` maps `authResult.reason` to a single-sentence string. Sample:

| Reason code | Current copy |
|---|---|
| `no_token` | "No token stored. Open the GAW ModTools popup and claim a rotation invite." |
| `fetch_failed` | "Worker unreachable: [error]. Check connectivity, try Force re-hydrate." |
| `whoami_status` | "Worker rejected token (HTTP [N]). Token may be expired or rotated -- claim a fresh invite." |

**Problems with current copy:**

1. The text conflates **diagnosis** with **action**. "Claim a rotation invite" is a result, not a step. Mods do not know what "claiming a rotation invite" means mechanically.
2. "Force re-hydrate" is an opaque label for a background action with no visible feedback loop until it either works or produces a one-liner failure message.
3. There are no numbered steps. A mod seeing the banner for the first time reads a sentence and then has to infer what to do.

### Proposed 3-step recovery wizard copy

Three failure modes, three wizard scripts. These replace the current single-sentence strings:

---

**Failure Mode A -- `no_token` / `short_token` (no stored credential)**

> Step 1 of 3: Click "Open ModTools popup" (button above). The popup opens.
> Step 2 of 3: In the popup, select "I have an invite LINK" OR "I have a token". Paste what your lead sent you.
> Step 3 of 3: Click Save. If the popup says "Welcome, [username]", you're done -- this banner will close.
> Still stuck? Your invite link may be expired. Ask your lead for a fresh one.

**Failure Mode B -- `fetch_failed` / `no_response` (worker unreachable)**

> Step 1 of 3: Click "Force re-hydrate" (button above). Wait 5 seconds.
> Step 2 of 3: If the banner still shows, check https://greatawakening.win in a new tab. If the site itself is down, the worker is also down -- wait and try again later.
> Step 3 of 3: If the site loads but the banner persists, open DevTools (F12) > Console and paste the red error to your lead. This is a worker outage, not an extension problem.
> Still stuck? The worker may be rate-limiting you. Wait 2 minutes and reload.

**Failure Mode C -- `whoami_status` (token rejected by worker, HTTP 401/403)**

> Step 1 of 3: Your token was rotated. Click "Open ModTools popup" (button above).
> Step 2 of 3: In the popup, go to Settings > Token and delete the current token.
> Step 3 of 3: Ask your lead for your current rotation token or a fresh invite link. Paste it in the popup. Click Save.
> Still stuck? You may be using a lead token in the mod slot (or vice versa). Ask your lead which token type you need.

---

### Implementation notes for Rule 99

The three wizard scripts above should be rendered as numbered `<ol>` elements inside the banner, not as flat `<div>` text. The current banner builds a single `<div>` with `textContent` -- the text cannot contain formatted steps. Change to innerHTML with a sanitized template or build the `<li>` nodes programmatically (no `innerHTML` on untrusted strings -- `reasonText` is hardcoded, so `innerHTML` is safe here).

The "Still stuck?" line at the end of each script is a Rule 99 requirement: recovery copy must anticipate the next failure mode, not leave the mod stranded at the end of step 3.
