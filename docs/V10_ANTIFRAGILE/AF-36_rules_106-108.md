# AF-36 -- Staged Testing Roadmap (Rules 106-108)

**Rules covered:**
- Rule 106: Unit tests simulating SW termination and message failures
- Rule 107: Property-based testing for all state transitions
- Rule 108: E2E tests with Playwright + extension loading in all failure modes

**Status:** AUDIT-ONLY -- no tests exist at v10.5.1. This document is the
implementation plan for v10.6 minimum-viable foundation.

**Codebase state at audit:**
- `background.js`: 2,428 lines. Service worker, RPC dispatcher, token vault,
  schema check, rate-limited clearTokens, weekly maintenance alarm.
- `modtools.js`: 23,192 lines. Content script. withUndo middleware, schema
  migrations (v1+v2), settings shape validator, user similarity signature,
  promote/demote team features, AI hold queue panel (SQ object), RPC relay.
- `popup.js`: 5,117 lines. Token onboarding, maintenance panel, relay
  surface for chrome.runtime.sendMessage.
- Zero test files. Zero package.json. Zero CI configuration.

```json
    "fast-check":               "^3.19.0",

    "fake-indexeddb":           "^6.0.0"
  }
}
```

**Why these choices:**
- **Vitest** -- native ESM, no transpile config, V8 coverage built in. Zero
  config overhead for a browser extension that ships vanilla JS. Jest would
  require a custom transform for the extension's non-module globals.
- **fast-check** -- property testing in JS with shrinking. Integrates with
  Vitest via a single `fc.assert(fc.property(...))` call in a normal `it()`.
- **Playwright** -- only framework with first-class MV3 extension loading.
  Chromium can load a packed/unpacked extension via `--load-extension` in the
  launch args. Selenium and Cypress cannot do this without hacks.
- **sinon** -- for SW lifecycle stubs (terminate, resurrect, alarm fire).
  Vitest's built-in mocks handle module mocking; sinon handles timer faking
  and the `chrome` global's event emitters reliably.
- **fake-indexeddb** -- background.js uses chrome.storage.local (mocked via
  chrome-mock.js) but popup.js talks to IDB via a utility. fake-indexeddb
  gives a real IDB API in Node without a browser.

## 2. Highest-Impact Unit Tests (5-10)

Each test is grounded in a real code path identified in the audit.

### U-1: Token Shape Regex (`__isValidTokenOrEmpty`)

**Why highest impact:** This is the vault's only guard against malformed token
writes. A regression here means a CLI-injection-ready token (`-flag-value`)
slips into secretCache and gets passed to wrangler/curl via debug paste.
Three acceptance criteria from v9.3.15 must hold simultaneously.

```
File: tests/unit/token-shape.test.js
Source: background.js L296-306 (__TOKEN_SHAPE_RE + __isValidTokenOrEmpty)

Test cases:
  PASS: 32-char alphanumeric "aB3cD4eF5gH6iJ7kL8"*2
  PASS: 256-char max-length string
  PASS: empty string (explicit clear)
  FAIL: 31 chars (below floor)
  FAIL: 257 chars (above ceiling)
  FAIL: leading dash "-abc123def456..."
  FAIL: leading underscore "_abc123def456..."
  FAIL: trailing dash "abc123def456...-"
  FAIL: all-digit string (no letter)
  FAIL: all-alpha string (no digit)
  FAIL: all-dash or all-underscore (both caught by letter+digit requirement)
  FAIL: non-string input (null, number, object)
  FAIL: string containing space or special char ($, @, !)
  BOUNDARY: exactly 32 chars valid content -- passes
  BOUNDARY: exactly 256 chars valid content -- passes
```

This test is a pure function import. Extract `__isValidTokenOrEmpty` to a
named export in background.js or inline the logic in the test via module
re-use. Zero chrome.* API dependencies.

### U-2: `withUndo` Idempotency and TTL Tier Assignment

**Why highest impact:** withUndo wraps every destructive action (ban, remove,
bulk-remove, DR-rule-add). A bug in clientOpId generation or TTL tier
selection would silently mis-configure the undo window for the most
dangerous actions in the system. The v11 universal undo mandate (Cat 4 Bet 2)
makes this load-bearing for the entire v11 surface.

```
File: tests/unit/with-undo.test.js
Source: modtools.js L6554-6583

Stubs required:
  - crypto.randomUUID() -- vitest globalThis mock
  - _setUndoSlot(), _showUndoToast(), _gamUndoAnnounce() -- sinon stubs

Test cases:
  1. Tier A call: ttlMs === 20000 in the slot written to _setUndoSlot
  2. Tier B call: ttlMs === 5000
  3. Missing tier defaults to 'A' (L6555: `const tier = opts.tier || 'A'`)
  4. actionFn throws: snack called with 'error', slot NOT written
  5. actionFn returns {ok: false}: slot NOT written, toast NOT shown
  6. actionFn returns {ok: true}: slot written, toast shown, result returned
  7. Two sequential calls: two distinct clientOpIds (UUID uniqueness)
  8. clientOpId fallback (crypto.randomUUID not available): format matches
     'undo-<timestamp>-<alphanum>' regex
```

### U-3: User Similarity Signature (`_computeSignature`)

**Why highest impact:** The Firehose V10.3 user similarity feature
(`02_USER_SIMILARITY.md`) uses _computeSignature to hash post/thread bodies
for lookalike detection. A regression in the kind dispatch or the
sha1-12-hex truncation means false positive or missed lookalike hits that
the AI hold queue then surfaces. This is an upstream data quality gate for
the AI pipeline.

```
File: tests/unit/user-similarity.test.js
Source: modtools.js L5441-5456 (_computeSignature)

Test cases:
  1. kind='User': returns lowercased id string directly, no hashing
  2. kind='Thread': hashes first 5 whitespace-split tokens of subject
     -- same subject -> same hash (determinism)
     -- subject with >5 tokens: only first 5 contribute
     -- empty subject: consistent (no throw)
  3. kind='Post': hashes first 80 chars of body
     -- body truncated at exactly 80 chars before hashing
     -- body < 80 chars: uses full body
  4. kind='QueueItem': same behavior as 'Post'
  5. kind unknown: falls back to lowercased id (L5455)
  6. opts null or seedData missing: no throw, returns id fallback
  7. Two calls with identical inputs: identical outputs (pure function)
  8. Two calls with different bodies: different hashes (collision resistance
     at 12-hex-char scale is probabilistic -- test corpus designed to avoid)
```

### U-4: Schema Migration Idempotency (`runMigrations`)

**Why highest impact:** runMigrations runs on every content-script load.
If migration 1 (roster 'pending' -> 'new') or migration 2 (hideSidebar
force-ON) re-fires on a schema-current install, it overwrites mod preferences
on every page load. This was the exact failure mode that forced the
K_SCHEMA gating.

```
File: tests/unit/schema-migration.test.js
Source: modtools.js L4308-4385

Stubs required:
  - localStorage (fake via Object.defineProperty or vitest fake timers pattern)
  - lsGet / lsSet (extract or stub)
  - setSetting (stub)
  - chrome.storage.local (chrome-mock.js)

Test cases:
  1. Fresh install (schema=0): both migrations run, hideSidebar set true,
     any roster entries with status='pending' renamed to 'new'
  2. schema=1 present: migration 1 skipped, migration 2 runs
  3. schema=2 present (current): both migrations skipped, localStorage not
     written, setSetting not called
  4. schema > SCHEMA_VERSION (future-schema on downgrade): neither migration
     runs (current < 1 is false for any stored value >= 1), localStorage
     stays at the stored version -- no regression
  5. Idempotency: call runMigrations() twice from schema=0; setSetting called
     exactly once (not twice) because after first call schema is 2
  6. Corrupt schema (NaN, 'garbage', null): parseInt fallback to 0, runs
     all migrations safely
  7. validateSettingsShape called after migration: shape check fires for
     the chrome.storage.local stub -- test that missing keys are reported
     in the result object
```

## 3. Highest-Impact Property Tests (3-5)

Property tests are not about specific inputs -- they are contracts that must
hold for ALL inputs in a domain. fast-check generates hundreds of random
cases and shrinks failures to minimal counterexamples automatically.

### P-1: State Machine Transitions for Tier Promote/Demote

**Contract:** The feature toggle promote/demote state machine must be
consistent. For any sequence of promote and demote calls, the final state
of `isTeam` must equal the value set by the last operation in the sequence.
No sequence of interleaved promote/demote calls should leave the state
undefined or stuck.

```
File: tests/property/state-machine.prop.test.js
Source: modtools.js L9989-10061 (promote/demote block)

fast-check arbitraries:
  - fc.array(fc.oneof(fc.constant('promote'), fc.constant('demote')),
             {minLength: 1, maxLength: 20})
  -- generates a random sequence of up to 20 operations

Property:
  For any sequence ops:
    initialState = { isTeam: false }
    finalState = ops.reduce(applyOp, initialState)
    expected = ops[ops.length - 1] === 'promote' ? true : false
    assert finalState.isTeam === expected

Failure mode this catches: a promote that doesn't flip isTeam because
the snack/RPC stub throws and the state update is inside the catch block
-- specific inputs might pass but the property reveals the partial update.
```

### P-2: AI Hold Queue Claim Race Safety

**Contract:** Two concurrent modAction calls claiming the same queue item ID
must result in exactly one successful claim (claimed_by = one mod) and one
409 conflict. This is the core atomicity guarantee of the UPDATE...WHERE
claimed_by IS NULL pattern.

```
File: tests/property/ai-hold-queue-claim.prop.test.js
Source: docs/V10_V11/02_AI_HOLD_QUEUE.md B1 + B2 (worker SQL)
        Implemented as: worker logic simulation with in-memory store stub

fast-check arbitraries:
  - fc.nat({max: 50})  -- number of concurrent mods (1..50)
  - fc.nat({max: 100}) -- number of queue items (1..100)
  - fc.array(fc.nat(), {minLength: 1, maxLength: 10}) -- claim attempts per mod

Property (tested against a JS simulation of the SQL UPDATE WHERE NULL pattern):
  For any N mods simultaneously calling claim(itemId):
    - Exactly one mod receives claimed_by = their username
    - All other N-1 mods receive a 409 conflict response
    - After all claims settle: exactly one row in 'claimed' state per item

Secondary property:
  For any sequence of claim(id) then resolve(id) then claim(id):
    - Second resolve on same id returns 409 (already resolved)
    - resolved_at is immutable once set (idempotent resolve)

Note: This tests the JS simulation, not the live D1. The E2E tests (E-7
below) cover the real database path.
```

### P-3: Undo/Redo Invariants

**Contract:** For any sequence of withUndo-wrapped actions and undo
invocations, the following invariants must hold:
1. An undo can only be executed once per clientOpId (slot consumed on use).
2. An undo fired after TTL expiry must be a no-op (returns without calling
   inverseFn).
3. An action returning {ok:false} never installs an undo slot.

```
File: tests/property/undo-redo-invariants.prop.test.js
Source: modtools.js L6554-6583 + _setUndoSlot, _executeUndo

fast-check arbitraries:
  - fc.record({
      tier: fc.oneof(fc.constant('A'), fc.constant('B')),
      actionSucceeds: fc.boolean(),
      undoCalledAt: fc.nat({max: 25000}),  -- ms after action
      undoCalledTwice: fc.boolean()
    })

Properties:
  1. If actionSucceeds=false: inverseFn call count === 0, regardless of
     undoCalledAt or undoCalledTwice
  2. If actionSucceeds=true AND undoCalledAt < ttl: inverseFn called exactly 1
  3. If actionSucceeds=true AND undoCalledAt >= ttl: inverseFn called 0
  4. If undoCalledTwice=true AND first undo succeeded: second undo is no-op
     (slot cleared after first execution)
  5. Tier A (ttl=20000) allows undo at t=19999, denies at t=20000
  6. Tier B (ttl=5000) allows undo at t=4999, denies at t=5000
```

### P-4: Token Shape: Complement Properties

**Contract:** For any string that passes __isValidTokenOrEmpty, mutating it
in specific ways must always produce a reject. This is the inverse of U-1 --
we're not testing the regex against fixed cases but proving structural
properties of the validator.

```
File: tests/property/state-machine.prop.test.js (extend P-1 file)
Source: background.js L296-306

fast-check arbitraries:
  - Generate a valid token: fc.stringMatching(/^[A-Za-z][A-Za-z0-9]{30,254}[0-9]$/)
    (guaranteed to pass: starts with letter, ends with digit, has both)

Properties tested on mutations:
  1. Prepend '-' to valid token -> always rejected (leading dash)
  2. Prepend '_' to valid token -> always rejected (leading underscore)
  3. Append '-' to valid token -> always rejected (trailing dash)
  4. Strip all digits -> always rejected (no digit)
  5. Strip all letters -> always rejected (no letter)
  6. Truncate to 31 chars -> always rejected (below floor)
  7. valid token repeated until >256 chars -> always rejected (above ceiling)

This catches regex refactors that accidentally widen the acceptance window.
```

## 4. Highest-Impact E2E Tests (5-10)

All E2E tests load the real unpacked extension via Playwright's
`--load-extension` arg. No mocking of chrome.* APIs. The GAW worker is
stubbed via Playwright's network interception (`page.route()`) so tests
are hermetic and do not require a live Cloudflare deployment.

### E-1: Fresh Install -> Auth -> First Ban -> Reload -> Still Authed

**Why highest impact:** This is the most important regression: the auth
carryover bug (`AUTH_CARRYOVER_RCA_2026-05-08.md`). If secretCache rehydrates
incorrectly after SW eviction, the mod is silently de-authed after a reload
and their next ban returns 401 with no visible error to the user.

```
File: tests/e2e/fresh-install-auth.spec.js

Steps:
  1. Launch Chromium with extension loaded into fresh profile (no prior data)
  2. Navigate to https://greatawakening.win/ (real page or local mock)
  3. Assert: extension toolbar icon is present, badge is empty
  4. Open popup via page.click('[title="GAW ModTools"]') or
     await extensionPage.goto(popupUrl)
  5. Enter a valid-shaped token in the workerModToken field
  6. page.route('**/mod/whoami', r => r.fulfill({body: JSON.stringify(
       {ok:true, data:{username:'test_mod', is_lead:false}})}))
  7. Click Save / submit token
  8. Assert: popup shows 'Authenticated as test_mod'
  9. Close popup
  10. Force SW eviction: await browser.serviceWorkers()[0].evaluate(
        () => self.registration.unregister())  -- Playwright SW handle
  11. Reload the GAW page
  12. Re-open popup
  13. Assert: token field still populated (not blank), whoami badge still shows
      username (re-auth fired from session storage, not from scratch)
  14. Simulate a ban: page.route('**/mod/ban*', r => r.fulfill({body:
        JSON.stringify({ok:true})}))
      Trigger ban UI on a post element
  15. Assert: ban modal sends correct Authorization header (inspected via
      page.on('request', ...) -- not 401)
```

### E-2: SW Kill Mid-Action (Rule 106)

**Why highest impact:** Rule 106 explicitly names this. The SW can be killed
by Chrome at any point. If it dies mid-RPC the content script hangs forever
on chrome.runtime.sendMessage (no sendResponse fires). The extension must
surface this to the mod as a recoverable error, not a silent hang.

```
File: tests/e2e/sw-kill-mid-action.spec.js

Steps:
  1. Load extension, authenticate (same as E-1 steps 1-8)
  2. Intercept a ban RPC call with a 2-second delay (simulate slow worker)
  3. While the call is in-flight, terminate the SW:
       const [sw] = await context.serviceWorkers();
       await sw.evaluate(() => self.registration.unregister());
  4. Assert within 5 seconds: the content script shows an error snack
     ("Connection lost" or equivalent) rather than hanging indefinitely
  5. Optionally: verify the UI recovers -- a second ban attempt after SW
     restart succeeds (SW is re-registered on next message)

Failure mode this catches: no timeout on chrome.runtime.sendMessage in the
RPC relay. The content script waits for sendResponse forever, the mod thinks
the ban is processing, and moves on to the next action without knowing it failed.
```

### E-7: withUndo -- Ban -> Undo -> Verify Unban RPC

**Why:** The most user-visible antifragile feature. Rule 106 names "message

the original action.

## 7. Known Constraints and Risks

**MV3 + Playwright maturity:** Playwright's extension loading works in
Chromium but requires a persistent browser context (not the default ephemeral
context). The `context.serviceWorkers()` API to get a handle on the
background SW is available from Playwright 1.36+. Confirm the installed
version supports it before starting E-2.

**SW eviction simulation:** Chrome does not expose a
"terminate this specific SW" API from JavaScript. The Playwright workaround
is `sw.evaluate(() => self.registration.unregister())` followed by
navigating to trigger re-registration. This is not a true mid-flight kill
(which requires CDP `Target.activateTarget` then closing the tab). E-2 may
need a CDP fallback if the unregister approach does not produce a mid-RPC
kill reliably.

**modtools.js is not modular:** At 23k lines, modtools.js uses no ES modules.
Unit tests for functions inside the IIFE require either (a) extracting the
function to a named export, (b) using Vitest's `vm` module to evaluate the
file and capture globals, or (c) testing via the exported
`window.__GAM_validateSettingsShape` surface that already exists. The
`withUndo` and `_computeSignature` functions will need minimal extraction
before U-2 and U-3 can import them. This is a deliberate constraint -- do
not refactor the whole file to make testing easier; extract only what is
needed for the test, leave the rest in place (Rule: surgical changes only).

**No real worker in CI:** All E2E worker calls are mocked via
`page.route()`. This means E2E tests verify the client-side state machine
and UI but do not validate the D1 SQL claim-race atomicity (P-2) against a
real database. That validation requires a staging Cloudflare environment
and is out of scope for v10.6.
