## Rule 115 — Semantic Versioning (Strict Compliance)

**Finding: PARTIAL COMPLIANCE. Version history is broadly correct but underdocumented; convention gaps create ambiguity for future releases.**

### Version History Review

| Version | Type claimed | Justified? |
|---|---|---|
| v9.24.0 | Feature release (minor bump) | Yes — last release in the v9 series before the MV3/architecture rewrite. |
| v10.0.0 | Major bump | **Requires verification — see analysis below.** |
| v10.5.0 | Feature release (minor bump) | Yes — changelog entries show new capabilities (AI providers, Discord retry queue, hot-path indexes) without breaking API changes visible to content-script consumers. |
| v10.5.1 | Patch release | Yes — changelog entry is purely bug fixes (token state persistence, DOM ID escaping, Firehose auth). |

### Was v10.0.0 a Real Major Bump?

Semver defines a major version bump as: any change that breaks backward compatibility for consumers of the public API. For a browser extension, the relevant "public API" is:

1. The content-script ↔ background message protocol (type strings, payload shape)
2. The `chrome.storage` key schema consumed by popup.js and modtools.js
3. The worker API surface consumed by the extension (endpoint paths, auth headers, response shapes)

The v9 → v10 transition coincides with the MV3 (Manifest V3) migration — moving from a persistent background page to a service worker. This is a structural breaking change: any code that assumed a persistent background context (long-lived in-memory state, persistent message ports, unguarded `chrome.extension.getBackgroundPage()` calls) would fail on v10. The manifest itself changed from `manifest_version: 2` to `manifest_version: 3`. **The major bump was justified.**

However, the CHANGELOG does not document this rationale. The v10.0.0 entry (if it exists outside the current CHANGELOG.md window) should have stated: "Breaking: migrated to MV3 service worker. Background page eliminated. All persistent state moved to chrome.storage.local/session. Any v9 sideloaded builds are incompatible."

**Recommendation R115-1:** Backfill a v10.0.0 CHANGELOG section (even retroactively) with the breaking-change statement. Future developers reading the log need to understand why the major number incremented.

### Was v10.5.0 a Feature Bump?

Yes. The 8.3.0 worker release (2026-04-25) and the corresponding client changes added net-new capabilities (AI fallback chain, circuit breaker, Discord retry queue) without removing or incompatibly changing existing endpoints or storage keys. A minor bump is correct.

### 3-Part vs. 4-Part Version Numbers

Chrome extensions support both `"version": "10.5.1"` (3-part) and `"version": "10.5.1.123"` (4-part, where the 4th segment is typically a build number). The current manifest uses 3-part. **This is the correct choice for this project.** Rationale:

- The 4th segment (`build`) carries no semantic meaning under semver and creates confusion when comparing versions (is `10.5.1.2` newer than `10.5.1.1`? By build number, yes — but semver comparison tools treat both as `10.5.1`).
- The Chrome Web Store treats the 4th segment as a build counter for auto-versioning pipelines; since GAW ModTools is sideloaded (not CWS-published), the 4th segment adds complexity with no benefit.
- Document the decision explicitly: **GAW ModTools uses 3-part semver (`MAJOR.MINOR.PATCH`). 4-part versions (`MAJOR.MINOR.PATCH.BUILD`) are not used. The build number, if needed for CI tracing, is recorded in release metadata only — never in the manifest.**

### Proposed Stricter Convention

```
MAJOR  Incremented when the extension breaks backward compatibility:
       - manifest_version change (MV2 → MV3)
       - storage key schema removal or rename
       - background message protocol breaking change (type rename, payload restructure)
       - worker endpoint removal or incompatible signature change

MINOR  Incremented for backward-compatible new capabilities:
       - new popup sections, new content-script features
       - new worker endpoints or optional fields added to existing ones
       - new storage keys (additive, existing keys preserved)

PATCH  Incremented for backward-compatible bug fixes only.
       No new features. No storage schema changes.
       Test: "a mod who doesn't reload their tab would see no behavioral difference."
```

**R115-2:** Add this convention table to `docs/VERSIONING.md` (new file) or prepend it to CHANGELOG.md. The definition of "breaking change for this extension" is non-obvious; writing it down prevents minor-for-breaking bumps in future releases.

## Rule 116 — In-Extension Changelog

**Finding: VIOLATION. CHANGELOG.md exists on disk but is not surfaced in the extension UI. Mods see no "what changed" communication post-update.**

### Current State

`CHANGELOG.md` is a developer-facing file at the repo root. After the extension auto-updates (via the `gam_update_available` banner flow), there is no user-visible "what's new" surface. The popup shows the version number in `#ver` (`pop-ver` span, currently hardcoded to `v5.2.8` — a stale value that should be set dynamically via `chrome.runtime.getManifest().version`), but no changelog link.

### Proposed Implementation

**Two-part solution: (A) auto-show on first popup open after update, (B) persistent link for on-demand access.**

#### A. `chrome.runtime.onInstalled` Trigger (background.js)

The `onInstalled` listener at `background.js:216` already fires on `update`. Extend it:

```js
// Inside the existing chrome.runtime.onInstalled.addListener handler,
// after the alarm re-creation block:
if (details.reason === 'update') {
  await chrome.storage.local.set({
    gam_show_whats_new: chrome.runtime.getManifest().version
  });
}
```

#### B. popup.js — Auto-Open on First Popup-Open After Update

On popup init, check the flag and open the "What's New" panel:

```js
const { gam_show_whats_new } = await chrome.storage.local.get('gam_show_whats_new');
if (gam_show_whats_new === chrome.runtime.getManifest().version) {
  await chrome.storage.local.remove('gam_show_whats_new');
  openWhatsNewPanel();  // see below
}
```

The flag is a version string (not a boolean) so it is idempotent: if the popup is opened twice before the flag is cleared, only the first open triggers the panel. Race-safe because `chrome.storage.local.remove` is async and the popup is single-instance.

#### C. "What's New" Panel

Implement as an in-popup modal overlay (consistent with existing modal patterns in popup.js). Content is a static JS object — not a fetch from CHANGELOG.md — to avoid CSP violations and offline failures:

```js
const WHATS_NEW = {
  '10.5.1': [
    'Token state now persists reliably across tab refresh and SW restart.',
    'Fixed lead-token writes clobbering team-token state.',
    'Fixed Settings panel render crash on dotted feature keys (e.g. features.drawer).',
    'Firehose auth now uses the correct canonical token getter.',
  ],
  '10.5.0': [
    'AI fallback chain: caller specifies preferred provider; worker falls back in stable order.',
    'Claude 3.5 Haiku added as third AI provider.',
    'Discord retry queue (D1 table) with exponential backoff and on-demand drain.',
    'Hot-path D1 indexes for actions and precedents tables.',
    'CORS lockdown on admin and bot endpoints.',
  ],
};
```

The panel title: "What's new in v10.5.1". Show the current version's entries plus the previous version's (two versions visible). "Dismiss" button clears the overlay. A persistent "What's new" link in the popup footer opens the same panel on demand.

#### D. Fix the Stale Version Display

`popup.html` line 24 hardcodes `v5.2.8`. This should be set dynamically in `popup.js` on init:

```js
document.getElementById('ver').textContent =
  'v' + chrome.runtime.getManifest().version;
```

This is a separate 1-line bug fix that should ship regardless of the changelog panel.

### Implementation Scope (AUDIT-ONLY — not implementing here)

Files requiring changes:
- `background.js` — add `gam_show_whats_new` write in `onInstalled` handler
- `popup.js` — add flag check on init, `openWhatsNewPanel()` function, `WHATS_NEW` constant
- `popup.html` — add modal markup (or generate it dynamically in popup.js), add footer "What's new" link
- `popup.css` — modal overlay styles (likely already have a pattern; reuse it)

No new files required. No manifest changes required (the `chrome.storage` key is already permitted).

## Rule 117 — Browser Version Testing

**Finding: VIOLATION. No `minimum_chrome_version` in manifest. No documented test matrix. `chrome.action.openPopup()` requires Chrome 127+ but runs on unguarded paths.**

### Minimum Chrome Version — What It Should Be

The extension uses Manifest V3, which requires Chrome 88+. However, several APIs used narrow this floor significantly:

| API | Minimum Chrome version |
|---|---|
| MV3 service worker | 88 |
| `chrome.action` (replaces `chrome.browserAction`) | 88 |
| `chrome.storage.session` | 102 |
| `chrome.offscreen` (not used, but common MV3 need) | 109 |
| `chrome.action.openPopup()` | **127** |
| `AbortSignal.timeout()` (used in fetch calls) | 103 |

**`chrome.action.openPopup()` at Chrome 127 is the binding constraint.** The codebase already has a feature-detection guard at `background.js:933` (`typeof chrome.action.openPopup === 'function'`), which is correct. But without `minimum_chrome_version` in the manifest, Chrome will silently load the extension on versions that will silently fail this check.

**R117-1:** Add to `manifest.json`:
```json
"minimum_chrome_version": "116"
```

Rationale for 116 rather than 127: `chrome.action.openPopup()` already has a runtime feature-detect fallback. Chrome 116 is the safe floor for all other used APIs including `chrome.storage.session` (102), `AbortSignal.timeout` (103), and the full MV3 service worker stability window. Mods on Chrome 116-126 lose the auth-fail banner → auto-open popup shortcut only; everything else functions. Mods on Chrome <116 should be blocked outright — the failure modes there are unpredictable.

**R117-2:** Document the 127 requirement for `chrome.action.openPopup()` in the feature flag guard comment in background.js. Current comment (line 926) mentions Chrome 127 but is developer-facing only; add a note that the `minimum_chrome_version: 116` floor means this guard will fire for ~5% of users on 116-126.

### Proposed Test Matrix

| Browser | Channel | Priority | Notes |
|---|---|---|---|
| Chrome | Stable (current: ~125) | P0 | Primary target. Full regression on every release. |
| Chrome | Beta | P1 | Catch API removals/changes before they hit stable. Run smoke test on release cut. |
| Chrome | Previous stable (n-1) | P1 | Mods rarely update Chrome immediately. Must not break on one version back. |
| Brave | Stable | P1 | Chromium-based; shares the Chrome engine but has aggressive ad-blocking that can interfere with content scripts on some sites. Test that `modtools.js` injection is not blocked. |
| Edge | Stable | P2 | Chromium-based. MV3 support matches Chrome. Worth a basic smoke test; unlikely to diverge. |
| Chrome | Dev | P3 | Early warning on API deprecations. Run opportunistically, not on every release. |

**What "smoke test" means at minimum:**
1. Extension loads without service worker errors.
2. Popup opens; version number displays correctly; Stats tab populates.
3. Token onboarding flow completes (enter token, save, reload — token persists).
4. Content script injects on `greatawakening.win`; mod console appears.
5. `chrome.action.openPopup()` path: on Chrome 127+ the auth-fail banner button opens the popup; on 116-126 the snack fallback appears.

**R117-3:** Add a `docs/BROWSER_TEST_MATRIX.md` documenting this matrix formally, with a column for "last tested version" that gets updated on each release. This creates an audit trail and prevents the "it should work on Edge" assumption from compounding silently across releases.

### Brave-Specific Risk

Brave's Shields may block `fetch()` calls from content scripts to the worker domain (`gaw-mod-proxy.gaw-mods-a2f2d0e4.workers.dev`) depending on the user's Shields setting. This is not a Chrome API compatibility issue — it's a Brave-specific network interception. The fix (if it surfaces) is user-side (add the worker domain to Shields allowlist), but the extension should detect a fetch failure and surface a clear error rather than silently hanging. The audit-found gap: current fetch error handling in `modtools.js` returns generic "network error" messages that do not distinguish a Brave Shields block from a genuine worker outage. This is a separate issue (outside Rule 117 scope) flagged here for awareness.
