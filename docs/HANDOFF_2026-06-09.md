### 3b. Why the lockout is now structurally prevented (v10.28.0 already self-heals)

Confirmed by reading the code this session — the current build recovers a token WITHOUT user action:
- **`loadSecrets()` (background.js:388)** runs on `onInstalled` AND `onStartup` (survives SW
  eviction): session → local (plaintext-first, decrypt fallback) → **L5 backup self-heal** (restores
  from a decrypt-independent backup if the vault came up empty; mirrors back to `gam_settings`).
- **`HEALTH_ALARM` every 5 min** (background.js:52, 1959) → `_healthCheck()` (1489) calls
  `loadSecrets()` → proactively re-heals the vault from durable storage on a timer.
- **L4** keeps the backup fresh whenever a valid token is held.
- Content-script HUD reads the durable local token directly (`getModToken()` →
  `getSetting('workerModToken')` → `_allSettings()` ← chrome.storage.local).

So a service-worker eviction — the thing that kept locking Commander out — is a **non-event on
v10.28.0**. The only residual lockout vectors are rare + user-initiated: full data wipe
(uninstall / "clear all browsing data") or server-side token rotation. Both recover in ~20s.
