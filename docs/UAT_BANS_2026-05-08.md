## B) BROKEN (wiring gaps + bugs)

1. **`ban-preflight` / `ban-confirm` worker endpoints are ORPHANED.**
   `gaw-mod-proxy-v2.js:11773-11774` registers
   `/mod/ban-preflight`, `/mod/ban-confirm` and `:2980–3110` implements
   the rate-limit + chained-audit + kill-switch + orphan-sweep machinery.
   **The extension never calls either endpoint.** `Grep` over
   `D:\AI\_PROJECTS\modtools-ext\*.js` returns ZERO matches for
   `ban-preflight|banPreflight|ban-confirm|banConfirm` (only docs).
   `modtools.js:7644` calls `apiBan` directly (`/ban`) and never wraps it.
   **Impact:** the 10/min + 100/day per-mod ban quota does NOT apply, the
   `bans_disabled` lead kill switch does NOTHING, `correlated_action` audit
   correlation never fires, and the orphan-sweep on line `:3081` runs on an
   empty table. **Fix:** in `modtools.js:7637`, before `apiBan`, call
   `rpcCall('modBanPreflight', { target, duration_hours, reason })`; on 429/503
   abort with the worker's `retry_after_seconds` snack; on 200 capture
   `audit_id`; after `apiBan` returns, fire-and-forget
   `rpcCall('modBanConfirm', { audit_id, gaw_response_status: r.status })`.
   Add the two RPC tools to `background.js`.
2. **Ban modal passes `last_messages: []` to AI** —
   `modtools.js:7224`. The "Generate with AI" entry from the BAN modal
   never has thread context (because it's a ban, not a modmail), so the
   AI loses one of its three context inputs. The worker tolerates an
   empty array but the AI ends up writing without the user's actual
   words. **Fix:** for ban-modal context, populate `last_messages` with
   the offending content text from `evidenceText`/`getContentText(item)` —
   author=target, body=evidenceText.
3. **Stale duplicate worker file** —
   `D:\AI\_PROJECTS\modtools-ext\worker\gaw-mod-proxy-v2.js` (8,209 lines,
   last modified May 4) lacks ALL macros + summarize-ban + ai-reply-for-thread
   + ban-preflight handlers. The LIVE worker is at
   `D:\AI\_PROJECTS\cloudflare-worker\gaw-mod-proxy-v2.js`. Risk: an admin
   running `wrangler deploy` from the wrong path would silently downgrade
   to v9.4-era code, killing 4 of the 5 features in this UAT. **Fix:**
   delete the stale copy or symlink it to the live one.
4. **`apiBan` duration semantics ambiguous** — `modtools.js:7641–7643`.
   Permanent ban passes `daysForApi=0`; warning (`duration===0`) goes
   through a different branch `:7615` (modmail only). But the worker
   `ban-preflight` validation `:3052` REJECTS `duration_hours <= 0` —
   so even if (B1) is fixed, perma-ban hits 400. **Fix:** when wiring
   preflight, send `duration_hours: 24*365*5` for perma + flag
   `permanent: true`, and have worker special-case the perma path.
5. **AI summarize-ban budget hit on EVERY successful ban** —
   `modtools.js:7670` fires `aiSummarizeBan` unconditionally inside a
   try/catch. `aiPreflight` in worker `:4856` will 429 when budget is
   exhausted; the catch swallows silently and falls back to first-14-words
   `:7679`. Working as designed BUT no telemetry on AI fallback rate. Mods
   never know they've been silently downgraded. **Fix:** if
   `ar.data.error === 'ai_quota'`, snack a quiet "AI summary skipped (daily
   budget) — using local fallback" once per day.
6. **`mc-ban-modmail` checkbox does not survive duration-warning branch** —
   `modtools.js:7618` for `duration === 0` (warning) sends one modmail.
   `:7708` for ban-path checks `if (alsoModmail)` and sends a SECOND
   modmail. But the warning path silently ignores `alsoModmail`. The
   "Also send a separate modmail" label remains visible during warning
   selection — a mod could check it expecting a duplicate notice. **Fix:**
   hide/disable the modmail checkbox when `selectedDuration === 0`.

## C) UX TRAPS

1. **`window.prompt()` for macro Add/Edit** — `modtools.js:7179, 7181`. Two
   sequential `prompt()` calls (label, body) inside the ban modal. On
   Chrome these stack visually behind the modal; mods report missing the
   second prompt. Body prompt accepts up to 4000 chars but `prompt()`
   single-line input mangles newlines on paste. Real mods will type
   short macros only. **Fix:** convert to inline `<input>+<textarea>` row
   below the dropdown when `__add__` is selected.
2. **No "edit existing macro" UI in the ban modal** — `__add__` only
   creates. To edit, mod has to find the popup macro manager (offscreen).
   Background RPC supports id-based update via `macroUpsert` `:1247` but no
   UI surface in the BAN tab.
3. **No "delete macro" UI in the ban modal** — same; only the popup has it.
4. **AI ban-summary preview is decorative** — `modtools.js:7104–7109`
   renders `#mc-ban-summary-wrap` but it stays `display:none` (never
   toggled to visible). The "after BAN" parenthetical is never shown to
   the mod. They have no idea what's about to be added to notes until
   they look at the user's note tab afterward. **Fix:** stream the summary
   in *before* the ban fires (preview-only call) and let mod edit before
   send.
5. **"AI suggest" generates 4 candidates but the dropdown UI says "AI
   drafted N replies"** — `modtools.js:7235`. The grid only shows 2
   columns `:7238` `grid-template-columns:1fr 1fr`. With 4 replies the
   grid becomes 2x2 which works, but with 1-3 (worker degrade-mode line
   `:4830–4835`) the grid is misaligned.
6. **`mc-ai-preview` `<div>` survives modal close** — created at
   `modtools.js:7214` and `appendChild`-ed to `msgIn.parentNode` `:7218`
   but never removed when modal closes. New ban modal opens → multiple
   AI preview blocks accumulate.
7. **Ban-hammer with no detected user → snack error, no recovery hint
   beyond "click on a username"** — `modtools.js:6435`. On long /threads
   listing this misfires; mod has to scroll-hover-and-rebrowse. Fine as
   error message, fragile in practice.
8. **Repeat-offender escalation auto-multiplies duration without flagging
   the mod** — `modtools.js:7367` (`days = Math.min(days*3, 90)`). A
   3-day default suddenly becomes 9-day on second offense; the duration
   button updates silently. Add a yellow flash + tooltip "auto-escalated
   from N → 3N due to prior bans".
