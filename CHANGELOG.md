## v10.36.12 — P0 FIX: the /users burst banner now retires, and "Death Row all" never lies (2026-07-05)

**v10.36.12 fixes the USERS-page trust-break Commander named as P0** (extension manifest 10.36.11 → 10.36.12; no worker change): *"⚠️ Burst detected: 29 users from IP range x.x.x — Death Row all 29"* stayed on screen unchanged for weeks. He pressed "Death Row all 29," got no confirmation and no lasting trace, and called it *"lazy, bad design, plain pure and simple."* A 10-specialist audit + adversarial critic traced both root causes in code before any fix was written.

**Root cause 1 — the banner was immortal by construction.** `getIPClusters(users)` clusters every user sharing a public IP prefix and never excludes users already Death-Rowed, banned, cleared, or reviewed. The banner rebuilt from that same unfiltered map on every render, so actioning the 29 kept their IP hash and the identical cluster re-formed forever — there was no code path for the banner to ever shrink or disappear.

**Fix:** a new `getUnresolvedIPClusters(users)` sits alongside the original `getIPClusters` (left completely unchanged — its 3 other legitimate callers, the cluster view filter and risk-flagging, still need full membership history). It excludes anyone already `deathrow`/`banned`/`cleared`/`reviewed` before clustering. The burst-alert renderer and both of its click handlers ("Select all N" and "Death Row all N") now compute against the unresolved set, so a cluster that's been fully actioned drops below the 3-person raid threshold and the banner retires on its own — no dismiss button needed, no new persistence key.

**Root cause 2 — "Death Row all" was a silent no-op after the first click.** `addToDeathRow` is intentionally idempotent (72h delayed ban, undoable) — queuing the same 29 users twice always returned 0 newly-added the second time. `batchDeathRow` only ever counted that "newly added" number, so a repeat click (or a click on an already-partially-actioned cluster) produced a bare **"0 user(s) added to Death Row"** toast in yellow "warn" styling that auto-dismissed in ~4 seconds — a zero-valued, self-erasing, alarm-colored toast that reads exactly like "it didn't happen," even when it had already happened earlier.

**Fix:** `batchDeathRow` now tracks `added` vs `already` vs `total` and always tells the truth: *"29 queued for Death Row — auto-bans in 72h unless undone"* (success) when everyone is new, *"10 newly queued, 19 already on Death Row — all queued (72h)"* (success) on a mixed batch, and *"All 29 already on Death Row — nothing to do. Use 'Flush Death Row now' to execute immediately"* (calm blue "info," not yellow "warn") when nothing changed. There is no code path left that can show a bare "0 user(s)" toast.

**Deliberately NOT touched this pass (documented, not silently skipped):** the existing Tards-section "APPLY TO ALL" bulk bar was independently confirmed already success-toned and already passing `fromUserAction:true` per-row — it was not the broken path and was left alone rather than "fixed" a second time. Per-row undo tracking for bulk cluster Death-Row was also deliberately NOT added this pass: the undo stack caps at 10 entries, so wiring 29 simultaneous per-row undo timers would silently evict 19 of them from undo-ability and fire 29 near-simultaneous 20-second grace-window UI rebuilds — a real feature, but one that needs its own batch-undo primitive, named as v.next rather than shipped half-working.

**Edge case documented, not "fixed":** actioning a cluster shrinks the *banner* immediately (via the status-exclusion above), while the affected rows still visibly linger for ~20 seconds in the Unreviewed list under the existing grace-window undo affordance — both behaviors are intended and independent; a future reviewer should not try to "reconcile" them into one timing.

**Verified from my side (§8):** `node --check modtools.js` PARSE OK. Two new slice-eval smoke tests exercise the real functions verbatim: `scripts/_p7_banner_excludes_actioned_smoke_test.mjs` (7/7 — a 4-person cluster with 2 already Death-Rowed drops to 2 unresolved survivors and falls below the raid threshold; a 5-person all-new cluster still renders in full; `getIPClusters` itself is proven unchanged; a `reviewed:true` user is excluded even while `status` is still `'new'`; private IPs stay excluded) and `scripts/_p8_batchdeathrow_truthful_counts_smoke_test.mjs` (19/19 — 29 fresh users yields `added=29`, a re-run of the same 29 yields `added=0, already=29` with an info-toned "already on Death Row" message and never a bare "0 user(s)," and a mixed 10-fresh/19-already batch reports both counts correctly). Full existing suite (21 files now, 300+ total assertions) re-run clean — one pre-existing static assertion in `scripts/_select_all_cluster_smoke_test.mjs` was updated in the same commit to expect the new `getUnresolvedIPClusters` call (an intentional, documented behavior change from WS-1, not a regression).

## v10.36.7 — FIX: /users Unreviewed list was not chronological (2026-07-01)

**v10.36.7 fixes Commander's live report right after the v10.36.6 boot-crash hotfix: the /users Triage Console now mounts and shows real data, but "the users list is not presented in chronological order."** (extension manifest 10.36.6 → 10.36.7; no worker change.)

**Root cause:** `buildTriageData()`'s sort comparator had two tiers with two DIFFERENT and inconsistent ideas of "chronological." On-page (currently-visible) users were sorted by raw DOM scrape order (`_domIdx`) under a v5.2.9 comment claiming *"GAW renders `.log` elements newest-first on the /users page — that IS the chronological registration order."* Off-page (historical roster) users were sorted by `joinedAt` — a real timestamp already computed for **every** user, on-page included, in `buildUserRecord()` via `parseRelativeAge(joinText)` (the GAW-displayed "N units ago" text). On-page rows had this authoritative timestamp sitting on the object and never used it. Confirming this was a genuine regression rather than a stylistic call: `renderTriageList()`'s own 24h-divider logic (~L16510) already documents and depends on the assumption *"Items are sorted joinedAt-desc"* — the codebase's own downstream consumer contradicts the upstream DOM-order sort.

**Fix:** `joinedAt` (via the existing, already-trusted `parseRelativeAge`) is now the primary chronological sort key for BOTH on-page and off-page users. DOM scrape index / `lastSeen` remain as tiebreakers only when `joinedAt` is missing or ties — so the fix degrades gracefully to the old behavior on any row `parseRelativeAge` can't parse, rather than ever doing worse than before.

**Verified from my side (§8):** `node --check` PARSE OK. Full existing suite (15 files, 194 total assertions) re-run clean, zero regressions. Boot-crash probe (the harness from v10.36.6) re-run clean — no synchronous throw at top-level init.

**Not root-caused with certainty (documented, not hidden):** whether GAW's actual `.log` DOM order changed, was never newest-first, or ties into the still-open, previously-flagged gap that `spans[1]`/`spans[2]` (join-date, IP-hash) extraction has no fallback/validation. `joinedAt` sidesteps the question entirely by trusting GAW's own displayed relative-age text instead of DOM position — but if that text extraction is itself wrong on the current GAW markup, this fix inherits that risk. ## v10.36.6 — HOTFIX: `mc is not defined` — stray backticks in a CSS comment killed the entire content-script boot (2026-07-01)
