**v10.16.14 UPDATE BANNER STALE-VERSION FIX** (`5ece609`, 2026-05-12) — Real bug reported by Commander: *"ModTools update: you're on v10.16.1 → latest v10.16.13 — I update, but it doesn't reflect it."* **Root cause**: `VERSION` constant at modtools.js:38 is captured at content-script INJECTION time via `chrome.runtime.getManifest().version`. When the user reloads the extension via chrome://extensions, the OLD content script in already-open tabs stays alive with its captured `VERSION='v10.16.1'`, even though the extension is now v10.16.13. The banner text reads from this stale `VERSION` and persistently shows the wrong "you're on" version until the user hard-refreshes each tab — a step that's mentioned in the banner's hint text but easy to miss after the chrome://extensions Reload arrow click. The existing `_gamShowExtOrphanedBanner` orphan handler only fires when an RPC fails after the reload; if the user just updates and doesn't interact, the stale banner persists indefinitely. **Two-part fix**: (1) **New "↻ Reload page" button** added to the update banner — single-click `location.reload()` after the extension reload. Closes the missing step in the operator mental model. (2) **Orphan-state polling** — banner spins up a 3s `setInterval` that checks `chrome.runtime?.id`; when it returns falsy (extension has been reloaded → this content script is orphaned), the banner removes itself and hands off to `_gamShowExtOrphanedBanner()`, which renders the canonical "Extension was reloaded. Refresh this page to reconnect." banner. The handoff cleans up the interval timer on every banner exit path (close, recheck, orphan-detect) so no zombie timers. Net diff: ~30 lines (1 new button + cleanup helper + orphan-poll), single file.
